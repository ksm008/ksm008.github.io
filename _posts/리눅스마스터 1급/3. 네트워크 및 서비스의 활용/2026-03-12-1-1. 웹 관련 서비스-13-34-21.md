### 1. 웹의 개요

### 2. 웹의 구성요소
1. 웹 문서 (HTML)
	- HTML은 W2C에서 주관
	- 동적 문서는 PHP, 자바 서블릿, 파이썬 등.
	- 자바 스크립트랑 같이 정적 문서의 제한된 기능 보완 가능
2. 웹 서버
	- 아파치, IIS, Ngnix, GWS 등이 존재
		- Apache : 사용자의 요청에 따라 별도 프로세스, 혹은 스레드를 생성하여 처리. Loadable Module 기능을 제공하여 서버의 동작을 확장 가능
		- IIS : 마소에서 제공하는 웹 서버. 
		- NGINX : 비동기식. 로드 밸런스, HTTP 캐시, 리버스 프록시 등을 기본 제공.
		- GWS : 구글 제공 웹 서버
	- 처음에는 단순 웹 문서 전달만 했지만 지금은 동적 페이지 등등 다양한 기능을 자체적, 도는 모듈을 사용하여 제공함.

> 전체 서비스를 작은 단위의 독립적 서비스로 설계하고 상호 연동하는 방식으로 개발하는 것

3. 웹 브라우저
	- 사용자 요청을 웹 서버에 전달하고 문서를 표시함.
		- 파폭 : 모질라에서 만듬. 게코 엔진 기반

### 2. HTTP 프로토콜
- 1. HTTP 프로토콜
	- 개념 설명임
- 2. HTTP 주요 요청 메소드
	- GET : URL 형식으로 리소스 데이터를 요청
	- HEAD : 리소스의 헤더 정보를 요청
	- POST : HTTP 요청을 헤더와 바디로 구분하여 사용, 요청할 내용을 바디에 담아 전송함. 폼을 사용한 데이터 전송에 사용
	- PUT : 문서 만들거나 갱신
	- DELETE : 삭제
	- CONNECT : 프록시 같은 중간 서버 접속용
	- TRACE : 요청한 리소스가 수신되는 경로 확인용
	- OPTIONS : 웹 서버가 제공하는 메소드를 확인하기 위해 사용
- 3. 응답 코드
	- 대충 다음과 같다
		- 100번대 : 조건부 응답
		- 200번대 : 성공
		- 300번대 : 리다이렉션
		- 400번대 : 요청오류
		- 500번대 : 서버오류
	- 100번대
		- 101 : 컨티뉴
		- 101 : 프로토콜 변경
	- 200번대
		- 200 : 전송 성공
		- 201 : 생성됨
		- 202 : 허용됨
		- 203 : Non-authoriative information
		- 204 : No content
		- 205 : Reset Content
		- 206 : Partial Content
	- 300번대
		- 300 : Multiple Choices
		- 301 : Moved Permanently
		- 302 : Moved Temporary
		- 303 : See Other
		- 304 : Not Modified
		- 305 : Use Proxy
	- 400번대
		- 400 : Bad Request (클라 문법 오류)
		- 401 : Unauthorized (권한이 없어 인증 실패)
		- 402 : Payment Required (결재 필요 요청)
		- 403 : Forbidden (리소스에 대한 권한이 없어 인가 실패)
		- 404 : Not Found (존재하지 않음)
		- 405 : Method Not Allowed (해당 방식으로 요청 불가)
		- 408 : 타임 아웃
		- 아니 뭐 이렇게 자세하게 나오면 틀려야지 그냥
	- 500번대 
		- 500 : 서버 내부 오류
- 4. 헤더 구조
	- HTTP 프로토콜의 요청과 응답은 헤더, 바디로 구성됨.
	- 헤더와 바디는 `\r\n`으로 구분됨. 헤더와 바디 사이에 빈 줄이 있다는 뜻.
	- 헤더는 공통 헤더, HTTP 요청 / 응답 헤더. 총 3가지로 구분 가능.
	- 공통 헤더의 항목
		- Date, Connection, Cache-Control, Content-Type, Content-Encoding, Content-Length
	- 요청 헤더의 주요 항목
		- Method, URI, HTTP 버전, Accept, If-Modified_Since, User-Agent, Coocke, host, Origin, Referer
	- 응답 헤더의 주요 항목
		- HTTP 버전/응답코드, Server, Location, Set-Cookie, Expires, Allow

## 3. 웹 관련 서비스의 설치와 설정
### 1. 웹 서비스 구성을 위한 설치 목록과 고려사항
- **아파치 서버**
	- 1.x는 프로세스 기반 -> 접속자가 증가하면 시스템 성능 영향 줌
	- 2.x는 MPM 기반 (아파치 멀티 프로세싱 모듈).
	- MPM은 prefork랑 worker 두 가지 방식이 있음. 
		- prefork:  리눅스의 기본 설정, 하나의 프로세스가 하나의 쓰레드
		- worker : 프로세스당 여러 쓰레드, 동시 접속자가 많을때 적합
	- 아파치는 다양한 모듈을 적재할 수 있음. 원래 정적/동적 2가지지만 아파치 2.x는 동적 적재만 지원함.
	- 일반적으로 PHP랑 MySQL등을 한거번에 같이 설치함. 이를 LAMP라고 한다. 리눅스 + 아파치 + PHP + MySQL. 최근에는 MySQL대신 MarinaDB를 이용하기도 함
- **PHP**
	- 웹 서비스 로직 구현, 동적 웹 페이즈 생성을 위해 고안된 스크립트 언어.
	- 서버에서 실행되는 언어이며 동적 적재를 사용함. 다이나믹 셰어드 오브젝트.
- **MySQL**
	- 관계형 데이터베이스 관리 시스템.
	- C, C++, 자바, PHP 등 다양한 프로그래밍 언어를 위한 인터페이스를 제공한다.
	- 기본 포트는 **TCP/3306**을 사용.
	- PHP는 **/tmp/mysql.sock** 파일을 소켓으로 이용하여 MySQL에 접속한다.
- **웹 관련 서비스 컴파일 및 설치 시 고려사항**
	- 미리 루트로 로그인하는게 편할 것이다. `su -`
	- 기존에 설치되어 있을수도 있음.
		- `httpd -v` 또는 `rpm -qa | grep httpd`로 확인을 먼저 하기.
	- `pkill -9 httpd` : 아파치 강제 종료
	- 패키지 제거는 `yum remove httpd`
	- 아파치랑 PHP는 configure -> make -> make install 방식으로 소스코드를 컴파일 및 설치할 수 있다. 특히 configure 시 다양한 속성을 설정할 수 있는데, `./configure --help`로 세부내용을 볼 수 있음.
	- MySQL은 **cmake** -> make install
	- 일반적으로 아파치 -> SQL -> PHP순으로 설치하여 확인함.
	- 소스코드 설치와 패키지 프로그램을 사용하여 설치가 가능함.
		- 소스코드는 다운하면 **/usr/local/src**에 다운됨
		- 프로그램은 **/usr/local**에 다운됨

> **sudo와 dnf**<br>
> - sudo는 기본적으로 root의 권한을 사용함.
> - 최근 리눅스는 yum과 함께 dnf를 지원함. yum의 낮은 성능, 종속성을 개선한 패키지 관리 도구
> - sudo dnf install httpd 이런식으로 쓰면 됨.

### 6. 아파치 웹 서버의 구조와 세부 설정
- **웹 서버의 주요 디렉터리**
	- 소스코드 컴파일 식으로 설치하면 configure 설정에 따라 **/usr/local/apache**에 관련 파일이 설치됨.
	- **주요 디렉터리**
		- bin : 서버 실행 및 관리를 위한 프로그램
		- build : 컴파일 스크립트 파일 모음
		- cgi-bin : CGI 스크립트
		- conf
		- error
		- htdocs : 웹 페이지가 저장되는 기본 경로
		- icons
		- include
		- logs
		- man
		- manual : HTML 파일 형식 메뉴얼
		- modules
		- 뭐 대충 알잘딱 할수 있잖아?
	- 웹 서버를 패키지 관리자로 깔면 리눅스 시스템의 디렉터리 특징에 따라 관련 파일들이 저장됨.
	- 아파치 웹 서버의 주요 디렉터리
		- /var/www/html : 웹 페이지가 저장되는 곳
		- /bar/www/cgi-bin, error, icons : 대충 알잖아요
		- /var/logs/httpd : 로그 저장되는 곳
		- /etc/httpd/conf : httpd.conf 저장된 곳
		- /etc/httpd/conf.d : 아파치 웹 서버의 추가 설정 파일이 저장된 경로. autoindex.conf, userdir.conf, welcome.conf 등
		- /usr/lib64/httpd/modules : 모듈 저장 경로.
- **아파치 웹 서버의 실행 파일 목록**
	- ab : 벤치마킹 도구
	- apachectl : 웹 서버 실행, 종료, 재시작을 위한 도구. 설정 파일인 httpd.conf을 검사하기도 함.
	- htpasswd : 아이디, 암호를 생성
	- htdigest : 사용자 인증(다이제스트) 파일을 생성, 업뎃함.
	- httpd : 데몬 프로그램
	- logresolve : 로그 파일에서 아이피를 호스트로 변환하는 도구.
	- rotatelogs : 로그 로테이트를 위한 도구
- httpd 명령은 `httpd 옵션 파일이름` 형식을 따른다. 
	- -t : httpd.conf의 문법적 오류를 점검함.
	- -f : 특정 환경설정 파일을 지정하여 아파치 데몬을 실행
	- -S : 설정된 가상 호스트 목록을 출력
	- -l : 웹 서버와 함께 컴파일 된 모듈 목록 출력
	- -M : 로딩된 모든 모듈 목록 출력
- `apachectl 명령(스크립트)`의 세부 옵션은 아래와 같다.
	- start : 데몬 시작. apachectl start
	- stop
	- restart
	- configtest : 환경설정 파일 문법 점검
	- graceful : 현재 클라이언트의 접속을 유지하며 환경 설정 파일을 다시 읽어 구성함
	- status / fullstatus :  웹 서버의 상태를 출력함.
- **아파치 웹 서버의 환경설정 파일**
	- 아파치 웹 서버의 환경 설정은 **/etc/httpd/conf/httpd.conf** 파일을 기본 사용하나 아파치 2.2 이후 버전부터 관리의 편의성을 위해 별도 파일을 구성 가능. 별도 설정 파일은 **/etc/httpd/conf.d** 디렉터리에 저장됨.
	- 별도 설정 파일은 기본 설정 파일에서 Include 지시자로 포함할 수 있음.
	- 주요 설정 파일 목록은 대충 이름을 보면 알 수 있음.
	- httpd-dav : WebDAV(웹 서버에서 파일 목록의 조회, 수정, 삭제 등을 할 수 있는 프로토콜)를 설정하는 파일
	- 설정 파일 변경후에는 apachectl restart로 재시작한다.
- **httpd.conf를 이용한 설정**
	- 조오오온나 많다 진자. 씨발 너무 많아. 근데 무조건 나오는거
	-
