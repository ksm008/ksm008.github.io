## 1. 버퍼 오버플로우 공격
### (1) 버퍼 오버플로우 공격 개요
#### 1) 프로세스 메모리 구조와 스택 프레임 구조
- **(가) 프로세스 메모리 구조**
	- 프로세스 메모리 구조는 텍스트, 데이터, 힙, 스택 영역으로 구분됨.
	- **텍스트 영역**
		- 프로그램 코드와 상수가 정의됨. 
		- 읽기만 가능한 메모리 영역
		- 여기에 데이터를 저장하려고 하면 분할 충돌을 일으켜 프로세스가 중지됨.
		- **분할 충돌 (Segmentation Fault)**
			- 운영체제가 허락하지 않은 메모리 공간에 접근하려고 할 때 발생하는 메모리 보호 위반 상태를 말한다.
			- 읽기 전용인 텍스트 영역에 새로운 데이터를 쓰려고 하면 메모리 관리 장치(MMU)가 접근을 감지하고 운영체제에 보고함.
			- 그러면 운영체제가 규칙을 어긴 프로세스에 오류 시그널 (SIGSEGV, Segmentation Violation)를 보내서 강제로 종료시킨다.
	- **데이터 영역**
		- 전역 변수와 정적 변수가 저장되어 있음.
	- **힙 영역**
		- 프로그래머의 필요에 따라 동적 메모리 호출에 의해 할당되는 메모리 영역.
	- **스택 영역**
		- 함수 인자 값, 함수 내의 지역 변수, 함수의 반환 주소 등이 저장되는 영역.
		- 상위 메모리 주소에서 하위 메모리 주소로 데이터가 저장됨.
- **(나) 스택 프레임 구조**
	- 아래와 같은 구조로 이루어져있다.
		- Return Address for Main
		- Old Frame Pointer of Main
		- 매개 변수 1, 2...
		- Return Address for Calling Function
		- Old Frame Pointer of Calling Function
		- 지역변수 1, 2...
	- **Return Address**
		- 함수가 실행을 마치고 다시 되돌아갈 주소를 말한다.
		- 해커들은 이 반환 주소를 자신의 해킹 코드가 담겨있는 주소로 덮어씌운다.
		- 때문에 함수가 리턴하면서 해커의 코드 주소로 이동하게 되면 레지스터의 명령어 포인터가 해당 코드를 실행하게 된다.
	- **Old Frame Pointer**
		- 이전 프레임 포인터는 스택 기준점을 설정한다.

#### 2) 버퍼 오버플로우 공격의 기본 개념
- 버퍼 또는 데이터 저장 영역에 할당된 용량보다 더 많은 입력이 위치하면 다른 정보를 변경할 수 있는 조건.
- 이런 조건을 사용하여 공격자는 시스템을 중지시키거나 제어를 갖기 위한 특별한 코드를 삽입함.
- 프로세스가 정해진 크기의 버퍼 한계를 벗어나 이웃한 메모리 위치에 데이터를 겹쳐 쓰려고 시도하는 것과 같은 프로그래밍 오류의 결과로 발생함.

#### 3) 공격 원리
- 버퍼 오버플로우에 취약한 함수를 쓰는 예제 코드를 이용해 그 원리를 알아보자.
```c
int main(int argc, char *argv[]) {
	char buffer[10];
	strcpy(buffer, argv[1]);
	printf("%s\n", buffer);	
}
```
- 파일명은 `bugfile.c`이다.
- `bugfile abcd`와 같이 입력하면 결과값으로 abcd를 출력하는 간단한 프로그램.
- `argc`는 인수 개수, `*argv[]`는 인자로 입력되는 값에 대한 번지수를 저장한다.
	- `argv[0]`은 실행파일 이름, `argv[1]`부터 차례대로 인자의 내용이 들어간다.
- `char buffer[10]`으로 10바이트 크기의 버퍼를 할당해주었다.
- `strcpy(buffer, argv[1])`을 통해 버퍼에 첫 번째 인자를 복사한다. abcd 값을 버퍼에 저장한다는 뜻. 
	- 버퍼 오버플로우 공격이 일어날 수 있는 곳이다. `strcpy` 함수는 입력된 인수의 경계를 체크하지 않는다. 
	- 인수는 `buffer[10]`으로 10바이트 크기를 넘으면 안되지만 그보다 큰 인수를 받아도 스택에 쓰인다.
- `printf("%s\n", buffer)`로 버퍼에 저장된 내용을 출력한다.
- 대부분의 C언어 배열이나 문자열에 데이터를 순차적으로 채워 넣는 거의 모든 함수와 로직은 낮은 주소에서 높은 주소 방향으로 이동한다. 
	- `memcpy`, `gets`, `scanf`를 비롯해 `for`문으로 배열 인덱스(`buffer[0]`, `buffer[1]`...)를 증가시키며 값을 넣는 것도 전부 낮은 주소에서 높은 주소 방향.

### (2) 스택 버퍼 오버플로우
#### 1) 개요
- 버퍼 오버플로우의 공격의 90%는 스택 버퍼 오버플로우이다. 위에서 본 예제도 스택 버퍼 오버플로우.
- 보통 SetUID가 설정된 루트 권한의 프로그램을 공격 대상으로 함.
- 스택에 정해진 버퍼보다 큰 공격 코드를 삽입하여 반환 주소를 변경함으로써 임이의 공격 코드를 루트 권한으로 실행하도록 하는 방법.
- 1988년 모리스 인터넷 웜에서 처음 발견된 이후 계속 이용되어 왔다.
	- fingerd 데몬에서 사용한 gets() 함수의 검사되지 않은 버퍼 오버플로우를 사용하였다.

#### 2) 셸코드
- 버퍼 오버플로우의 공격 핵심은 오버플로우가 발생하는 버퍼에 저장되는 공격자의 코드로 실행 제어를 이동시키는 것이다. 
- 이 코드를 셸코드라고 한다.
- 이렇게 부르는 이유는 사용자 명령어 라인의 해석기인 셸로 제어를 넘기고 공격당한 프로그램의 권한으로 시스템의 다른 프로그램에 접근하기 때문이다.
- 즉 스택의 지역 변수 등에 기계어를 직접 집어넣고 해당 주소를 리턴 주소로 만들어 기계어 코드를 강제 실행시키는 방식.

#### 3) 스택 버퍼 오버플로우 공격 절차
1. 공격 셸 코드를 버퍼에 저장한다. 
2. 루트 권한으로 실행되는 프로그램의 특정 함수의 스택 반환주소 버퍼를 오버플로우 시켜서 셸 코드가 저장되어 있는 버퍼의 주소로 덮어씌움. 
3. 특정 함수의 호출이 완료되면 조작된 반환 주소로 셸 코드의 주소가 반환되어 셸 코드가 실행되고 루트 권한을 획득하게 된다.

### (3) 힙 오버플로우
#### 1) 개요
- 힙은 프로그램과 전역 데이터 위에 위치하며 메모리 위 방향으로 커진다. 스택과는 반대로 커짐. 스택은 아래로 커지니까...
- 힙에 요청되는 메모리는 레코드의 연결리스트와 같은 동적 데이터 구조를 위해 사용됨. 이런 레코드가 오버플로우에 취약한 버퍼를 가지고 있다면 연속된 메모리가 손상될 수 있음.
- 스택과 다르게 반환주소가 없어 실행 제어를 쉽게 이동할 순 없지만 할당된 공간이 함수에 대한 포인터를 포함하고 있다면 이 포인터 주소를 변경하여 겹쳐 쓴 버퍼에 있는 셸 코드를 가리키도록 할 수 있음.

### (4) 버퍼 오버플로우 공격 대응책
#### 1) 개요
- 크게 두 가지 방어법으로 분류될 수 있다.
- 컴파일 시간 방어 : 새 프로그램 내에서 공격을 저지하도록 프로그램을 강화하는 것을 목표
- 실행 시간 방어 : 존재하는 프로그램에서 공격을 발견하고 중지시키는 것을 목표로 함.

#### 2) 컴파일 시간 방어
- **(가) 기본 개념**
	- 컴파일 시간 방어는 프로그램을 컴파일 할 때 검사하여 버퍼 오버플로우를 방지, 발견하는 것을 목표로 한다.
	- 기본 대응책은 버퍼 오버플로우를 허용하지 않는 자바, ADA, 파이썬 같이 현대화된 고급 프로그래밍 언어나 소프트웨어를 작성하는 것이다.
	- C나 C++ 같이 메모리 접근에 높은 자유도를 가진 프로그래밍 언어를 사용하는 경우 데이터의 크기가 할당된 버퍼의 크기를 초과하지 않도록 반드시 확인하는 로직을 포함하여 안전하게 프로그래밍한다.
- **(나) 프로그래밍 언어의 선택**
	- 현대의 고급 수준의 프로그래밍 언어를 사용하여 프로그램을 작성하는 방법이 있다. 
		- 이러한 언어는 변수 타입과 그 타입에 허용되는 연산들에 대해 강력한 표기법을 제공한다.
	- 이런 언어들은 버퍼 오버플로우 공격에 취약하지 않다.
		- 컴파일러가 범위 검사를 강제로 수행하는 코드를 자동으로 추가하기 때문에 프로그래머가 명시적으로 그걸 코딩할 필요가 없다.
- **(다) 안전한 코딩 기법들**
	- 프로그래머는 코드를 검사하고 불완전한 코딩 구조를 안전한 방법으로 재작성할 필요가 있다.
	- **사용을 자제해야 하는 함수들**
		- strcat(), strncat(), strncpy(), gets(), scanf(), sscanf(), vscanf(), vsscanf(), sprintf(), vsprintf(), gethostbyname(), realpath() 등
	- **사용을 권장하는 함수**
		- strcat_s(), strncat_s 등 거의 다 뒤에 s가 붙는다.
		- gets_s() 또는 fgets()를 사용하도록 한다.
- **(라) 언어 확장과 안전한 라이브러리의 사용**
	- 표준 문자열 라이브러리를 안전한 것으로 교체하는 방법이 있다.
	- Libsafe가 잘 알려진 예시. 표준 의미를 구현하면서 추가적으로 복사 연산이 스택 프레임 내부의 지역 변수 공간을 넘어가지 않도록 검사하는 기능을 구현함.
	- 수정된 라이브러리는 최소한 표준 라이브러리만큼 효율적이라고 알려져 있으며 기존의 프로그램에 대한 버퍼 오버플로우 공격을 쉽게 방어할 수 있도록 해줌.
- **(마) 스택 보호 매커니즘(Stack Guard)**
	- 프로그램을 스택 오버 플로우 공격으로부터 보호하는 효과적인 방법 중 하나는 함수의 진입 종료 코드를 조사하고 스택 프레임이 손상이 있는지 검사하는 것이다. 어떤 변경이 발견된다면 공격을 허용하는 대신 프로그램을 종료시킨다.
	- Start Guard는 가장 잘 알려진 보호 매커니즘 중 하나. GCC 컴파일러의 확장 버전으로 추가적인 함수 진입과 종료 코드를 삽입한다.
	- 컴파일로가 프로그램의 함수 호출 시에 ret (반환 주소, return을 줄여 부름) 앞에 있는 canary(밀고자) 값을 주입하고 종료 시에 그 값이 변조되었는지 여부를 확인하여 버퍼 오버플로우 공격을 탐지한다.
	- 카나리아 값을 지역 변수와 반환 주소 사이에 넣는다. 버퍼 오버플로우가 발생하려면 무조건 이 카나리아 값도 덮어 씌워져야 한다는 점을 이용함.
- **(바) Stack Shield**
	- 함수 시작 시 복귀 주소를 Global RET라는 특수 스택에 저장해두었다가 함수 종료 시 저장된 스택의 RET 값을 비교해 다를 경우 오버플로우로 가정하여 프로그램 실행을 중단시키는 기술.

#### 4) 실행 시간 방어
- **(가) 기본 개념**
	- 대부분의 컴파일 시간 기법은 기존 프로그램을 다시 컴파일한다.
	- 기존의 취약한 프로그램에 보호를 제공하기 위하여 운영체제의 업데이트로 배포할 수 있는 실행 시간 방어법이 인기를 끌고 있음.
	- **실행시점 대응책**은 실행가능 주소 공간 보호 기법(Executable Address Space Protection)과 주소 공간 임의화(Address Space Randomization) 기법을 포함한다.
		- **실행가능 주소 공간 보호 기법 :** 실행 코드가 프로세스 메모리 상의 특정 위치에서만 실행될 수 있게 함으로써 공격자가 스택 버퍼에 주입한 실행 코드(셸코드)를 원천적으로 실행될 수 없게 만든다. 대부분의 버퍼 오버플로우 공격은 스택 버퍼에 실행 코드를 주입하므로 재컴파일 없이 기존 프로그램들을 보호하는데 매우 효과적.
			- 애초에 스택 안에서는 코드를 실행할 수 없도록 만든다.
		- **주소 공간 임의화 기법 :** 스택 버퍼가 위치하는 주소 공간을 메모리 내에서 임의적으로 배치하여 공격자가 스택 버퍼 속 주입한 셸코드의 주소를 예측할 수 없도록 함. 따라서 실행 코드로의 주소 반환 변경이 매우 어렵게 된다.
			- 스택 안의 셸코드를 실행하고 싶어도 주소가 매번 바뀌니 할 수가 없다.
- **(나) ASLR과 NOP**
	- **주소 공간의 임의 추출(ASLR, Address Space Layout Randomization)**
		- 공격을 막는 다른 실행 시간 기법으로는 프로세스 주소 공간에 있는 중요 데이터 구조의 위치를 조작하는 것이 있다.
		- 공격자는 셸코드로 제어를 넘기기 위해 공격에 사용할 적당한 반환주소를 결정하는데 예측된 주소를 사용한다. 이 예측을 어렵게 만드는 한 가지 기법은 각 프로세스 안의 스택이 임의의 다른 곳에 위치하도록 변경하는 것.
