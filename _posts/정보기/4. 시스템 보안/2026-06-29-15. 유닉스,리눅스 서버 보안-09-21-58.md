##  1. 유닉스(UNIX)
### (1) 개요
#### 1) UNIX 시스템의 특징
- 대화식 운영체제
	- 사용자와 운영체제 간의 상호작용은 명령어 해석기인 셸(Shell)에 의해 이루어짐
- 멀티태스킹
	- 여러 개의 명렁어를 동시에 처리하는 방식
- 멀티유저
	- 동시에 여러 사용자가 하나의 컴퓨터를 사용 가능
- 호환성 및 이식성
	- 하드웨어적인 구조에 독립적으로 응용프로그램을 작성할 수 있는 환경 제공
- 계층적 파일시스템 
	- 정보의 고유구조를 반영하도록 그룹화하는 기능으로 파일관리에 상당한 유연성 제공
- 뛰어난 통신 기능
	- TCP/IP에 기반한 통신기능을 운영체제 자체에 내장하고 있으며 프로세스 간을 연결하는 파이프 기능이 있어서 명령어 간의 데이터 전달이 수월
- 다양한 기능의 유틸리티와 개발환경 제공
	- UNIX 시스템은 수백여 개의 명령어와 유틸리티를 사용자 및 개발자에게 제공하여 도움을 준다.

#### 2) 유닉스 운영체제 종류
- 유닉스
	- 오라클 - 오라클 솔라리스
	- IBM - AIX
	- HP - HP-UX
- 리눅스
	- 레드햇 - 페도라
	- 캐노니컬 - 우분투

### (2) UNIX의 구성
#### 1) 커널(Kernel)
- **(가) 개요**
	- 커널은 항상 메모리에 상주하여 CPU, 메인 메모리, 하드디스크 등의 하드웨어 자원을 제어하면서 프로세스 스케줄링, 기억장치관리, 파일관리, 시스템호출 인터페이스(System Call), 입출력서비스 등의 기능을 사용자에게 제공함.
	- 커널은 가장 하위 수준에서 하드웨어와 직접 관계한다. 그러므로 커널의 일부분은 특정 컴퓨터 시스템의 하드웨어에 종속적이다. 그러나 유닉스는 현존하는 어떤 운영체제보다 다양한 종류의 컴퓨터에서 동작하도록 설계되어 있다.
- **(나) 시스템 호출**
	- 사용자 프로그램이 커널에 접근하는 인터페이스를 제공한다. 이러한 시스템 호출은 커널 내에 서브루틴 형태로 존재하고 있다.
	- 입출력 시스템 호출, 프로세스 간 통신시스템 호출, 프로세스 관리 시스템 호출 등 세 가지로 분류할 수 있다.

#### 2) 셸(Shell)
- 운영체제와 사용자가 대화하기 위한 기반을 제공하는 중요한 프로그램.
- 일반적인 운영체제에서 명령어 해석기(Command Interpreter)라고 불리는 부분의 기능을 담당한다.
- 유닉스에서 셸은 한 가지만 있는 것이 아니다. 최초의 셸인 본 셸(sh, Bourne Shell)을 비롯하여 C 셸(C Shell), 콘 셸(ksh, Korn Shell), 배시 셸(bash, Bourne Again Shell)등 다양하다.

#### 3) 파일시스템(File System)
- **(가) 디렉터리 구조**
	- 유닉스 파일시스템은 디렉터리와 파일로 구성되어 있다. 
	- 디렉터리는 계층화된 트리 구조를 가지며 최상위 디렉터리는 루트(root)이다.
	- 루트 디렉터리는 `/`로 표기되며 거기서부터 서브디렉터리가 계층적으로 분기되어 나간다. 각 디렉터리는 파일과 또 다른 디렉터리의 정보로 구성된다.
	- 대부분 유닉스 운영체제는 기본적으로 비슷한 시스템 디렉터리 구조를 가진다.
		- / : 루트 디렉터리나 최상위 디렉터리이다.
		- /etc : 시스템의 환경 설정 및 주요 설정 파일 등을 담고 있다. (passwd, hosts)
		- /dev : 프린터나 터미널 같은 물리적인 장치를 다루기 위한 특수 파일을 담고 있다.
		- /usr/bin : 기본적으로 실행 가능한 파일을 담고 있다. (echo, mv, copy, pwd 등)
		- /usr/include : C 언어 라이브러리 헤더 파일이 저장되는 디렉터리이다.
		- /usr/lib : 기본 프로그램의 모듈을 담고 있다.
		- /usr/sbin : 시스템 관리 명령어가 저장되는 디렉터리이다.
		- /home : 사용자 홈 디렉터리가 저장되는 디렉터리이다.
		- /tmp : 프로그램 실행 및 설치 시 생성되는 임시 파일을 담고 있다. 이 디렉터리에 파일을 저장하면 재부팅 시 임의로 삭제될 수 있음.
		- /var : 시스템 로그가 저장되는 디렉터리.
- **(나) 파일시스템의 구조**
	- 파티션에 생성된 파일 시스템은 부트 블록, 슈퍼 블록, i-node 리스트, 데이터 블록의 네 가지 영역으로 분리된 자료구조를 가진다.
	- 파일 시스템은 이들 네 가지 영역으로 구성된 자료구조를 제어함으로써 유지된다.
- **(다) i-node**
	- i-node(index node)는 유닉스에서 각 파일에 대한 정보를 기억하는 약 120바이트의 고정된 크기의 구조체이다. 
	- 일반 파일이나 디렉터리 파일의 i-node는 각 파일의 디스크 블록의 위치를 포함하고 있으며, 특수 파일의 i-node는 주변장치를 식별할 수 있는 정보를 포함하고 있다.
	- **i-node 리스트**
		- `ls -l`명령에 의해 출력되는 결과는 해당 파일의 i-node를 참조하는 것.
		- i-node에는 파일명이 없다. 파일명은 디렉터리를 통해 관리된다.
		- 침해 사고가 발생하게 되면 피해 시스템 파일에 대한 무ㅛdification, Access, Change) Time을 점검한다.
		- 특정 파일의 i-node에 대한 속성정보는 stat 명령을 통해 확인이 가능하다.
	- i-node는 파일에 대한 정보와 이 파일에 할당된 디스크 블록의 주소를 저장하는 필드들로 구성되어 있다.
	- **i-node의 속성**
		- 노드 번호 : 파일시스템 내에서 해당 파일을 식별하기 위한 고유한 식별자.
		- 파일 타입 : 일반, 장치 파일인지, 또는 디렉터리인지를 나타냄
		- 접근 권한 : 파일에 대한 접근권한
		- link count : 해당 노드를 참조하는 링크의 개수를 셈. 하드링크 기준.
		- 소유자 : 파일의 소유자/UID
		- 소유그룹 : 파일의 소유그룹/GID
		- 파일 크기 : 크기용
		- **MAC Time**
			- 마지막으로 수정된 시간, 마지막으로 파일에 접근한 시간, 마지막으로 파일의 속성을 변경한 시간을 기록한다. 여기서 파일의 속성은 i-node의 정보를 의미함.
		- Block Index : 데이터 블록(Data Blocks)에 저장되어 있는 파일 내용에 대한 색인 정보. 데이터 블록은 실제 파일의 내용이 저장되는 블록이다.
- **(라) 슈퍼 블록**
	- 파일시스템의 정보를 유지하는 자료구조. 
	- 좀 더 빠르고 효과적인 파일 시스템의 관리를 가능하게 한다.
	- 사용하지 않고 있는 디스크 블록과 i-node를 할당받기 위해서 i-node 리스트와 데이터 블록을 순차적으로 하나씩 비교하는 방식은 효율적이지 않음...
	- 따라서 슈퍼 블록에는 파일 시스템의 요약 정보와 함께 사용하고 않고 있는 i-node와 디스크 블록의 위치 정보도 가지고 있다.

## 2. UNIX 기본 사용법
### (1) 파일시스템 응용
#### 1) 파일과 디렉터리 관리
- **(가) 디렉터리(파일) 정보 출력(ls)**
	- 특정 디렉터리에 있는 디렉터리와 파일을 열거.
	- 추가 옵션으로 이들에 관한 여러가지 정보(크기, 소유주, 만든 시간, 변경 시간, 이름, 접근권한 등)도 알 수 있다.
	- **입력 문법 및 예문**
		- **옵션 리스트**
			- -l : 목록 형태로 파일의 정보를 자세하게 보여줌
			- -a : 도트(.) 파일을 포함하여 디렉터리 내에 있는 모든 디렉터리 및 파일을 보여준다.
			- -R : 하위 디렉터리에 있는 내용까지 보여준다.
			- -F : 파일의 종류를 알려준다. 디렉터리는 /, 실행파일은 `*`, 심볼릭링크는 @
			- -i : i-node 번호를 보여준다.
	- **파일 유형**
		- - : 일반 파일
		- d : 디렉터리 파일
		- l : 심볼릭 링크 파일
		- b : 블록 장치 특수 파일
		- c : 문자 장치 특수 파일
		- p : 파이프, 프로세스간 통신에 사용되는 특수 파일
		- s : 소켓, 네트워크 통신에 사용되는 특수 파일
	- **접근권한**
		- UNIX 운영체제에서는 권한 부여 대상을 세 종류로 구분함.
			- user : 파일을 만든 소유주
			- group : 파일을 만든 소유주가 속한 그룹
			- other : 기타 사용자
		- **접근권한의 종류**
			- 읽기 (r, 4) : 파일은 읽거나 복사가 가능하고, 디렉터리는 ls 명령으로 디렉터리 목록을 볼 수 있다. 이 때 ls 명령의 옵션은 실행 권한이 있어야만 사용이 가능하다.
			- 쓰기 (w, 2) : 파일은 수정, 이동, 삭제할 수 있고 (디렉터리에 쓰기 권한이 있어야 한다) 디렉터리는 파일을 안에 생성하거나 삭제할 수 있다.
			- 실행 (x, 1) : 파일은 실행할 수 있고 디렉터리는 파일을 디렉터리로 이동하거나 복사할 수 있다.
		- 유닉스 시스템에서 파일과 디렉터리를 생성할 때 파일의 기본 권한은 666, 디렉터리는 777이다.
		- 디렉터리가 777인 이유는 실행 권한이 없으면 안으로 들어가지조차 못하기 때문이다...	
	- 링크 카운터는 하드 링크수라고도 하며 동일한 i-node를 참조하는 파일의 수를 말한다.
	- **소유주**
		- 디렉터리 및 파일의 소유주를 의미하며 디렉터리 및 파일을 생성한 유닉스 시스템의 사용자 계정이 지정된다.
		- 소유주는 생성 후에도 변경이 가능하며 이때는 chown 명령을 사용한다.
	- **소유그룹**
		- 소속된 기본 그룹명이 지정됨. 소유 그룹은 생성 후에도 변경이 가능하며 이때는 chgrp 명령을 사용한다.

#### 2) 파일 권한 관리
- **(가) 접근권한 변경(chmod)** 
	- -R을 쓰면 하위 디렉터리와 파일의 권한까지 변경함.
	- 대상 : u(user), g(group), o(other), a(all)
	- 연산자 : +(추가), -(제거), =(지정)
	- 접근권한 : r(읽기), w(쓰기), x(실행)
		- 숫자로도 가능하다. 각각 4,2,1로 설정하고 싶은 권한 값을 합치면 된다.
		- 6은 4 + 2이므로 읽기, 쓰기 권한이다.
	- **예시**
		- `chmod go-w daemon.c` : daemon.c의 그룹, other에게 권한에서 쓰기를 제거함
		- `chmod g+x,o-x daemon.c` : daemon.c의 그룹에게는 실행 권한을, other에게는 쓰기 권한을 제거함
		- `chmod 777 daemon.c` : 모든 사용자에게 rwx 권한을 설정
		- `chmod 664 daemon.c` : 유저, 그룹에게는 rw, other에게는 r 권한만 설정
		- `chmod 600 daemon.c` : 유저에게만 rw 설정
- **(나) 소유권 또는 그룹 변경(chown/chgrp)**
	- 파일이나 디렉터리의 소유주, 그룹을 변경할 때 사용한다. 명령을 실행하고 나면 파일의 이전 소유주는 해당 파일에 이 명령을 다시 실행할 수 없다.
	- -R로 하위 디렉터리나 디렉터리 하위의 모든 파일의 소유주를 변경할 수 있다.
	- -h로 심볼릭 링크 파일 자체의 소유주나 그룹을 변경한다.
	- **예시**
		- `chown infosec daemon.c` : 소유주를 infosec로 변경한다
		- `chgrp other daemon.c` : 소유 그룹을 other로 변경한다.
	- 소유권과 관련된 명령어는 슈퍼 유저인 root만 사용 가능하다.

### (2) 프로세스 응용
#### 1) 프로세스 개요
- **(가) 첫 번째 프로세스**
	- 특정 프로그램을 실행하기 위해 새로운 프로세스를 생성하는 시스템 호출은 없다. 대신 유닉스에서는 기존의 프로세스를 복제한다. 그리고 생성된 자식 프로세스를 그 특정 프로그램과 연관시킨다.
	- pid가 0인 swapper라는 첫 번째 프로세스는 부팅 시간 동안 운영체제에 의해서 생성된다. 이 프로세스는 즉시 PID 1과 2를 가지는 2개의 프로세스 init과 pagedaemon을 만들기 위해 fork /exec를 두 번 실행한다.
	- swapper가 pid 0, init, pagedaemon이 각각 1, 2번이다.
	- 스와퍼와 페이지 데몬과 같은 프로세스는 그 중요성 때문에 커널 모드에서 영구적으로 실행된다. 이를 커널 프로세스라고 부른다.
	- 시스템 내의 모든 다른 프로세스들은 init 프로세스의 자손들이 된다.
- **(나) 프로세스의 생성 및 종료와 관련된 두 가지 규칙**
	- PID가 0인 프로세스를 제외한 모든 프로세스는 실행 중에 부모 프로세스를 가지게 된다.
		- 만약 부모 프로세스가 자식 프로세스보다 먼저 종료될 경우 자식 프로세스는 고아 프로세스가 되지만, 유닉스 시스템의 규칙에 따라 PID가 1번인 init 프로세스가 고아 프로세스의 대리모 역할을 한다.
	- 프로세스 종료 시 자신을 생성한 부모 프로세스에게 자신의 종료를 알린다.
		- 만약 부모 프로세스가 이를 확인하지 않았다면 종료된 자식 프로세스는 부모 프로세스가 확인해줄때까지 좀비 프로세스로 남는다. 
		- 시스템 커널의 입장에서 프로세스는 제한된 자원이므로, 좀비 프로세스가 과도하게 많아지면 더 이상 프로세스를 생성하기 못하게 된다.

> **좀비 프로세스**<br>좀비프로세스는 실행되고 있지 않지만 프로세스 테이블 목록을 차지한다. 만약 좀비 프로세스가 증가한다면 테이블이 꽉 차버려 정상 프로세스를 실행할 수 없다. 좀비 프로세스는 kill 명령으로도 제거할 수 없으며 부모 프로세스에게 SIGCHLD 시그널을 보내 부모 프로세스가 자식을 정리하거나 부모 프로세스 자체를 종료시켜야 한다. 부모 프로세스가 종료되면 좀비 프로세스는 고아가 되고, 고아 프로세스는 대리모 역할인 init이 자식의 종료 정보를 주기적으로 확인하고 정리한다.
{: .prompt-info }

## 3. UNIX 시스템 관리
### (1) 시스템 시작과 종료
#### 1) 부팅 관련 용어 정의
- **(가) 런 레벨(Run Level)**
	- 시스템에서 런 레벨의 의미는 시스템의 운영 상태를 숫자 혹은 문자로 표현한 것.
	- init 프로세스는 `/etc/inittab` 파일에 정의된 런 레벨에 따라 `/etc/rc.d/rd[x].d(여기서 x는 런 레벨)` 디렉터리에 나열된 스크립트를 실행하여 시스템의 운영상태를 구성한다.
	- 각 런 레벨이 의미하는 시스템의 운영상태는 다음과 같다.
		- 0 : PROM(Programmable Read-Only Memory) 모드
		- S, s : 시스템 싱글 유저 모드, 로컬 파일 시스템이 마운트 되지 않은 상태
		- 1 : 시스템 싱글 유저 모드, 로컬 파일시스템이 마운트 된 상태
		- 2 : 멀티 유저 모드(NFS 클라이언트 모드)
		- 3 : 멀티 유저 모드(NFS 서버 모드). UNIX 기본 런 레벨
		- 4 : 사용 안함
		- 5 : 시스템 power off 모드
		- 6 : 시스템 리부팅
- **(나) 시스템 런 레벨 단계 구성**
	- INIT의 상태는 런 레벨을 옮겨 다니며 각 레벨마다의 역할을 수행한다.
	- init 명령어 다음에 런 레벨을 쓰면 해당 런 레벨로 이동할 수 있다.
		- init 6, init 3 이런식.

#### 2) 시스템 종료
- **(가) 시스템 종료**
	- 운영중인 유닉스 시스템은 정기 점검, H/W 및 S/W의 추가 또는 변경, 기타 관리작업을 하기 위하여 시스템의 종료가 필요하다. 
	- 유닉스 시스템을 종료할 때 주의사항
		- 접속중인 사용자에게 시스템 종료를 공지하여 작업을 마무리 하도록 해야한다.
		- 운영중인 서비스(프로세스)를 안전하게 종료해야 한다.
		- 하드디스크를 갱신하여 파일시스템의 무결성을 유지한다.
	- 셧다운 명령은 시스템을 안전하게 종료할 때 사용하며 시스템의 런 레벨을 변경할 때도 사용한다.
- **(나) 하드디스크 동기화(sync)**
	- 유닉스 시스템은 하드디스크의 입/출력에 대한 효율성을 높이기 위하여 버퍼를 운영한다.
	- 시스템이 비정상적으로 종료되면 버퍼에 있는 데이터가 하드디스크에 반영되지 않아서 파일시스템에 무결성 문제가 생긴다. 따라서 하드디스크 동기화는 시스템을 종료하는 중에 해야 할 가장 중요한 작업이다.

### (2) 파일시스템 관리
#### 1) 파일시스템 연결(mount)
- 보조기억장치에 들어있는 데이터를 읽고 저장하려면 해당 장치를 유닉스 시스템에 연결해야한다.
- 마운트 명령을 통해 보조기억장치에 설치된 파일시스템을 유닉스 시스템이 인식하도록 특정 디렉터리에 논리적으로 연결시켜 준다.
- **예시**
	- `mount -a` : /etc/fstab 파일에 정의된 모든 파일시스템을 마운트한다.
	- `mount /dev/cdrom /mnt/cdrom` : /dev/cdrom 디바이스 파일(CD-ROM)을 /mrt/cdrom 디렉터리에 마운트한다.
	- 인수 지정 없이 `mount` 명령만 사용하면 현재 시스템에 마운트 된 정보를 출력한다.

#### 2) 파일시스템 연결 해제(unmount)
- 이전에 마운트 된 파일시스템의 연결을 해제한다.
- 프로세스가 파일시스템에 있는 파일을 사용 중이거나 사용자가 파일시스템의 디렉터리에 있다면 umount 명령은 `device is busy` 메시지와 함께 실패함.
- /etc/mtab 파일은 시스템에 마운트 된 파일시스템의 정보를 관리한다. mount 명령을 실행하면 파일시스템이 마운트 정보르르 /etc/mtab 파일에 기록하고 umount는 정보를 삭제한다.
- **예시**
	- `unmount -a` : -a는 마운트 된 모든 파일시스템을 언마운트한다.
	- `unmount -f` : -f는 해당 파일 시스템을 사용하는 프로세스를 강제로 종료하고 파일시스템을 언마운트한다.
	- `unmount /mnt/cdrom` : /mnt/cdrom 디렉터리에 연결된 파일시스템의 연결을 끊는다.
	- 루트 디렉터리 /는 언마운트할 수 없다.

### (3) 프로세스 스케줄 관리
#### 1) 정기적 스케줄 관리(cron)
- **(가) 개요**
	- 리눅스마스터 공부하면서 했던거잖아??
	- cron 데몬 프로세스는 유닉스 시스템에서 정기적인 작업을 지정시간에 처리하기 위해 사용한다.
	- cron 데몬 프로세스로 처리하는 것이 더 나은 프로세스는 다음과 같은 특징을 가짐.
		- 일괄적으로 처리해야하는 작업(특히 몰아서 하는 야간작업)인 경우
		- 작업에 대한 요구가 불규칙하지 않은 경우
	- cron 데몬 프로세스를 사용하려면 다음 세 가지 구성요소가 필요하다.
		- 정기적으로 처리할 작업 목록을 정의한 crontab 파일
		- crontab 파일을 제어하는 crontab 명령
		- crontab 파일을 읽어서 내용에 정의된 대로 작업을 처리하는 cron 데몬 프로세스
	- cron 데몬 프로세스는 유닉스 시스템에서 기본적으로 지원하는 데몬 프로세스이므로 사용자는 crontab 명령으로 작업목록을 정의하는 방법만 이해하면 된다.
	- **/etc/default/cron** 
		- CRONLOG 변수가 설정되어 있다.
		- 이 변수는 기본적으로 YES 값을 가지고 /var/cron/log 파일에 크론 작업에 사용된 명령어와 크론 작업이 시작한 시간, 끝난 시간이 로그에 남는다.
- **(나) crontab 파일의 구조**
	- crontab 파일은 정기적으로 처리해야 하는 작업목록을 정의한 파일이다. 작업목록은 cron 데몬 프로세스에 의해 처리된다.
	- crontab 파일의 각 행은 Space나 Tab으로 구분된 6개의 필드로 이루어짐.
		- `[분] [시간] [일] [월] [요일] [실행할 명령어]`
		- `,`로 개별 시간을 등록할 수 있음. 만약 시간 필드에 `9, 18`이라고 되어있으면 9시, 오후 6시에 명령을 수행함.
		- `-`로 특정 시간동안 수행하는 것도 가능함. 만약 시간 필드에 `09-18`로 되어있으면 9시부터 오후 6시까지 명령을 수행함.
		- `*`가 들어가면 매번을 뜻함.
		- `/`는 간격을 의미함. 만약 분 필드에 `*/10`이라고 되어있으면 매번 10분마다를 뜻함.
		- 앞에 `#`이 있으면 주석 처리로 실행하지 않음.
		- 월, 요일은 1부터 7까지 숫자로 나타내기가 가능. 시작은 1이며 월요일. 
		- 문자열로도 가능함. 월은 jan, feb... 요일은 sun, mon... 이런 형태.
		- **예시**
			- `01 24 * 1 Monday /home/francis/baskup.sh` : 매년 1월 월요일마다 23시 1분에 backup.sh를 실행함. 년도 필드에 `*`가 붙어있기 때문에 매년임.
			- `10 4 1 */2 * `: 2개월마다 1일 4시10분 실행
	- `crontab -l`로 안 내용을 출력할 수 있음.
- **(다) crontab 파일의 제어**
	- crontab 파일은 사용자 계정별로 만들어진다.
	- 시스템관리자인 root는 login_name을 명시적으로 사용하여 다른 사용자의 crontab 파일을 편집할 수 있지만 일반 사용자는 자신 파일만 편집할 수 있다.
	- **주요 옵션**
		- -e : 설정을 편집함
		- -r : 설정을 삭제함
		- -u : 특정 사용자를 지정. 사용하지 않으면 자신의 계정을 의미한다.
		- -l : 목록을 확인함
		- e 옵션을 쓰면 vi 에디터가 열린다.
	- 그 외에 /etc/cron.allow에는 사용을 허가하는 사용자 목록, .deny에는 제한하는 사용자 목록이 존재한다.

## 4. UNIX 서버 보안
### (1) 시스템 보안
#### 1) 사용자의 패스워드 관리
- **(가) 개요**
	- 유닉스 시스템을 이용하는 사용자의 계정과 인증을 관리하는 것.
	- /etc/passwd 파일은 유닉스 시스템에 등록된 사용자 계정마다 정보가 담겨 있는 파일.
	- 과거에는 패스워드를 인코딩한 텍스트가 passwd 두번째 필드에 담겨있었지만 최근에는 /etc/shadow 파일에 암호화된 패스워드를 저장한다.
- **(나) passwd 파일**
	- 일반적으로 모두 열람이 가능함. 하지만 쓰는건 루트나 권한이 있어야 함.
	-  구조는 다음과 같다. 
		- `로그인아이디 : 비밀번호 : uid : gid : 사용자 설명 : 홈 디렉터리 : 셸`
		- 로그인아이디 : 사용자 계정 또는 로그인 이름. 영문자와 숫자로 이루어지며 시스템마다 조금씩 다르지만 사용에 제약이 있다. 루트 계정은 시스템에 대한 총괄 권한을 가지고 있는 계정으로 공격자의 목표가 될 수 있으므로 원격 접속을 금지하는 것이 보안상 안전함. 일반적으로 루트 계정을 su(switch user)를 허용하는 계정 그룹을 생성한 후 해당 계정으로 접속 후 root 계정으로 전환하여 사용한다.
		- 비밀번호 : 리눅스나 이전 버전에는 이 자리에 비밀번호가 직접 기록되어있었다. 하지만 보안의 중요성이 커지면서 비밀번호를 별도로 저장하는 shadow 방식을 사용할 때 필드는 x로 표시하고 비밀번호는 root만 읽을 수 있는 /etc/shadow 파일에 저장한다.
		- uid (사용자 ID) : 유닉스는 내부적으로 각 사용자마자 일련번호를 부여해 구분한다. UID는 시스템이 부여하며 100번 이하는 시스템이 사용하는 값이고 0번은 시스템 관리자를 나타낸다.
			- UID는 중복되어 사용될 수 있다. 따라서 임의로 생성한 계정의 UID가 0으로 설정되어 있다면 root 권한 탈취를 위한 행위로 의심해봐야 한다.
			- GID도 동일하다.
		- gid (그룹 ID) : 사용자가 속한 그룹의 ID이며 일련번호로 되어있다.
		- 사용자 설명 : 사용자 관련 기타 정보로 일반적으로 사용자 이름을 설정한다.
		- 홈 디렉터리 : 로그인 후 사용자가 위치할 홈 디렉터리의 절대 경로이다. 루트는 /root, 일반사용자는 /home 디렉터리 하위에 위치한다.
		- 셸 : 로그인 셸의 절대 경로이다. 사용자가 로그인 셸의 종류를 바꾸어 달라고 요청하면 /etc/passwd 파일의 이 필드를 직접 수정하면 된다. 로그인이 불필요한 계정에 대해서는 로그인을 금지하는 것이 보안상 안전하다.
			- 로그인을 금지하려면 /sbin/nologin 또는 /bin/false로 설정한다.
- **pwconv, pwunconv, grpconv, grpunconv**
	- shadow 사용 유무를 결정함. 
	- 그냥 conv는 passwd를 가지고 shadow를 만듬. unconv는 shadow를 지워버리고 passwd에 암호화된 값을 집어넣음. 

```bash
cat /etc/passwd
root:x:0:0:root:/root:/bin/bash
...
```

>**/etc/passwd 파일 공격**<br>해킹 시 주로 UID, GID를 0번으로 변경한다. 시스템은 UID와 GID를 기준으로 권한을 부여하기 때문에 계정 이름이 root가 아니라도 UID, GID가 0번이면 root가 되는 것이다. <br>또한 로그인 셸을 변경하여 사용자가 로그인 시 악성 셸이 실행되게 하는 경우도 있으므로 주기적으로 변경 여부를 확인해야 한다.
{: .prompt-info }

- **(다) shadow 파일**
	- 패스워드 정보를 평문으로 저장하면 당연히 유출 피해가 잘 일어나니까 암호화해서 보호해야한다.
	- /etc/shadow 파일에 암호화된 패스워드가 저장되고 관리자만 읽을 수 있도록 제한한다. shadow 파일의 경우 root만 읽을 수 있도록 접근권한이 설정되어 있다.
	- shadow 파일에는 계정별 암호화된 패스워드 정보와 패스워드 에이징 정보가 저장되어 있다. 패스워드 에이징 정보란 시간의 흐름에 따른 패스워드 관리정책을 말한다.
	- 구조는 다음과 같다.
		- `계정 이름 : 암호화된 패스워드 : 마지막 변경일 : 패스워드 최소 사용일 : 최대 사용일 : 만료 경고일 : 유예기간 : 만료일`
		- 계정 이름 : 사용자 계정을 인덱스로 하여 /etc/passwd 파일과 사용자 정보를 서로 연계함.
		- 암호화된 패스워드 (일방향 해시 알고리즘 이용)
			- 형식 : `$id$salt$hashed` 형태. id는 암호화 알고리즘, salt는 소금값, hashed는 패스워드 해시값.
			- id는 암호화 알고리즘을 나타낸다. 
				- 1 : MD5
				- 2a, 2y : Blowfish
				- 5, 6 : 각각 SHA-256/512
			- 소금값은 패스워드 암호화 강도를 높이기 위한 값이다. 공격자의 사전 공격에 대응할 수 있는 소금값은 랜덤한 값으로 생성된다.
			- 패스워드 해시값은 패스워드에 소금값을 추가하여 암호화된 패스워드이다.
			- 리눅스의 경우 해당 필드의 값이 다를수도 있다.
				- null : 패스워드가 설정되어있지 않으므로 바로 로그인이 된다. 패스워드를 설정하도록 해야한다.
				- `*` : 패스워드 잠김 상태로 패스워드 로그인은 불가능 하지만 SSH 인증 등 별도의 인증을 사용하여 로그인이 가능하다.
				- !! : 패스워드 잠김 상태로 모든 로그인이 불가능하며 사용자 계정을 생성하고 패스워드를 설정하지 않으면 잠금 상태로 되어있다.
		- 마지막 변경일 : 1970년 1월 1일을 기준으로 며칠이 지났는지 그 일수를 표기한다.
		- 패스워드 최소 사용일 : 해당 기간동안은 무조건 설정한 패스워드를 사용해야 한다. 최소 사용기간을 설정하지 않으면 최근 암호 기억을 무력화하여 익숙한 패스워드를 재사용할 문제점이 있으므로 1일, 1주로 설정할 것을 권장.
		- 최대 사용일 : 최대 사용 기간을 설정하지 않으면 패스워드 유출 발생 시 계속 접근이 가능하므로 90일로 설정할 것을 권장한다.
		- 만료 경고일 : 최대 사용일에 지정한 일수가 얼마 남지 않았을 때 알리는 필드이다. 
		- 유예기간 : 시스템에 로그인을 자주 하지 않는 사용자의 경우 사용자 계정을 잠기게 한 후 로그인이 되지 않도록 하는 기능.
		- 만료일 : 1970년 1월 1일을 기준으로 며칠이 지났는지 그 일수로 표기한다.

#### 2) 접근권한 마스크(umask)
- 시스템 관리자는 /etc/profile에 umask를 설정 가능하다. 
- umask는 전체 사용자에게 기본 권한을 지정하기 위해 사용한다.
- 파일을 생성할 때는 666에서 umask 값을 빼고 디렉터리를 생성할 때는 777에서 뺀다.
- umask를 터미널에 단독으로 입력하면 설정된 umask 값을 출력하고, 뒤에 권한을 입력하여 설정할 수 있다.
- `umask 222` : 파일을 생성하면 666 - 222 = 444가 기본 권한이 된다.
- **umask 관련 설정 파일**
		- /etc/default/login
		- /etc/profile
		- `$HOME/.profile`
		- `$HOME./cshrc`

#### 3) 권한 상승(SetUID, SetGID)
- **(가) 개요**
	- 계정이 누구인지 식별하는 UID와 GID는 각가 RUID, RGID 라고도 한다. Real을 붙인 것. 하지만 유닉스에서는 어떤 권한을 가지고 있는가에 대한 UID, GID가 별도로 존재한다. 
	- 앞에 Effective를 붙여 EUID, EGID라고 부른다.
	- RUID/GID는 프로세스를 실행시킨 사용자의 ID
	- EUID/GID는 프로세스가 실행중인 동안에만 부여되는 ID. 자원에 대한 접근권한을 판단하기 위한 ID이다.
	- 최초로 로그인할 때는 R/EUID와 R/EGID는 같은 값을 가진다. 달라지는 때는 오직 SetUID 비트를 가진 프로그램을 실행했을 때만 프로세스 안에서 잠시 일치하지 않는 상태가 된다.
	- 때문에 SetUID 비트가 설정된 파일은 모두 해킹 대상이 될 수 있다. 백도어, 버퍼 오버플로우 등 여러 공격에 이용된다. 그래서 SetUID 비트가 설정된 파일을 목록화하여 관리하는 작업이 필요하다.
- **(나) SetUID 비트를 이용한 bash 셸 획득**
```c
#include <stdio.h>
main() {
		setuid(0);
		setgid(0);
		system("/bin/bash");
}
```
- **분석**
	- 프로그램이 실행되었을 때는 이미 root 권한의 프로세스이기 때문에 `system("/bin/bash");` 명령을 실행하는 프로세스의 권한은 root가 된다.
- **(다) SetUID 설정 시 주의사항과 SetGID**
	- SetUID가 설정되어 있는 프로그램을 실행하는 경우, 이 프로세스가 갖는 권한은 프로그램을 실행한 사용자의 권한이 아닌 프로그램 소유주의 권한이 된다.
	- 루트만 접근 가능한 파일이나 명령에 일반 사용자로 접근하는게 필요하면 SetUID를 적용하는게 시스템 운영면에서 효율적이다.
	- 루트 권한이 필요없는 프로그램에 소유주가 루트로 되어있으면서 SetUID가 설정된 경우 보안에 허점을 초래하게 된다.
	- 이런 이유로 시스템 관리자는 주기적으로 SetUID가 설정된 프로그램을 확인할 필요가 있다.
	- SetGID와 SetUID는 기능이 유사하다. SetGID가 설정된 명령 또는 프로그램은 실행 중인 프로세스의 EGID를 소유주의 그룹 ID로 변경한다. 
	- SetGId가 설정되어 있다면 실행중인 프로세스는 사용자가 속한 그룹이 아닌 소유주의 그룹 권한을 갖게 된다.

#### 4) 디렉터리 접근권한(sticky-bit)
- 유닉스 시스템은 sticky-bit를 이용하여 디렉터리에 특별한 접근권한을 부여할 수 있다. 그래서 sticky-bit를 공유모드라고 한다.
- 스티키 비트가 설정된 디렉터리는 시스템에 있는 모든 사용자가 파일이나 하위 디렉터리를 생성할 수 있지만 해당 디렉터리를 지우는 것은 소유주나 루트인 경우에만 가능하다. 주로 /tmp 같은 공용 디렉터리에서 자주 사용한다.
- 아래 표의 특수권한 파일검색 명령어에서 특수권한 8진수는 7000, 6000과 같이 더할수도 있다.

| 구분                | 기호방식 | 8진수 방식 | 특수권한 설정                                                                               | 특수권한 파일검색                    |
| ----------------- | ---- | ------ | ------------------------------------------------------------------------------------- | ---------------------------- |
| **setuid(4)**     | s    | 4000   | `chmod 4777 a.out`<br>`chmod u+s a.out`                                               | `# find / -perm 4000 -print` |
| **setuid(2)**     | s    | 2000   | `chmod 2777 a.out`<br>`chmod g+s a.out`                                               | `# find / -perm 2000 -print` |
| **sticky-bit(1)** | t    | 1000   | `chmod 1777 /testdir`<br>`chmod o+t /testdir` (Linux)<br>`chmod u+t /testdir` (SunOS) | `# find / -perm 1000 -print` |

### (2) 네트워크 보안
#### 1) 슈퍼 서버(inetd 데몬)
- **(가) 개요**
	- 유닉스 시스템에는 다양한 종류의 서버 프로그램이 실행되고 있다.
	- inetd 데몬은 N개의 개별 서버를 하나로 통합하여 클라이언트로부터 서비스 요청이 올 때마다 해당 서비스와 관련된 실행 모듈을 실행해준다. FTP, Tenlet, TFTP 등이 해당된다.
		- 스탠드얼론 방식 : 개별 서비스 별로 서버 프로세스(데몬)이 동작하는 방식이다. 속도가 빠르지만 서버 자원을 많이 점유한다.
		- inetd(xinetd) 방식 : 슈퍼 데몬을 사용하여 개별 서비스를 동작시킨다. 상대적으로 속도가 느리지만 서버 자원을 많이 점유하고 있는 단점이 있다.
	- 시스템에서 불필요한 서비스를 제한하기 위해 살펴봐야 하는 파일은 **/etc/inetd.conf** 파일이다. 불필요한 서비스는 `#`을 붙여 주석처리하면 제한할 수 있다.
- **(나) inetd.conf 파일**
	- 파일 구조는 다음과 같다.
	- `[서비스] [소켓 타입] [프로토콜] [대기 설정] [로그인 이름] [서버] [인자]`
	- `telnet stream tcp6 nowait root /usr/sbin/in.telnetd in.telnetd`
		- 서비스 : 서비스 이름이다. **/etc/services**에 정의가 되어있어야 한다.
		- 소켓 타입 : 서비스에 대한 소켓유형을 설정한다. TCP일 경우 stream, UDP일 경우 dgram으로 표기한다.
		- 프로토콜 : **/etc/protocols**에 정의된 프로토콜의 종류와 번호다. 임의로 정해진 번호가 아닌 표준 번호이다.
		- 대기 설정 : inetd가 클라이언트에 서비스를 요청받은 경우 즉시 또 다른 요청 처리를 할 것인지 여부에 따라 wait, nowait으로 구분된다. 
			- 다음의 접속 요청을 새로운 것으로 시작하는 것은 nowait
			- 기다리면서 다음 접속 요청을 다루는건 wait
			- TCP는 반드시 nowait, UDP는 wait이어야 한다.
		- 서버 : 해당 서비스를 수행하기 위해 어떤 프로그램을 실행할지를 절대 경로로 작성한다.
		- 인자 : 데몬을 실행하는데 필요한 인자 값을 적는다.

#### 2) 접근통제(TCP Wrapper)
- **(가) 개요**
	- 외부에서 들어오는 클라이언트에 대해 접근통제 기능을 제공한다.
	- 클라이언트의 IP 주소를 확인하여 시스템관리자가 접근을 허용한 호스트들에 대해서만 서비스를 허용하기 때문에 외부의 해킹으로부터 시스템을 보호할 수 있다. 
	- TCPWrapper가 설치되면 inetd 데몬은 TCPWrapper의 데몬인 tcpd 데몬에 연결을 넘겨준다. tcpd 데몬은 접속을 요구한 클라이언트가 적절한 접근 권한이 있는지를 확인하고 해당 데몬을 넘겨준다. 이 때 연결에 대한 로깅도 실시함.
	- TCPWrapper가 모든 프로토콜에 대한 접근제어를 할 수는 없음. 
		- inetd 데몬이 관리하지 않는 스탠드얼론 데몬
		- TCP 외의 일부 프로토콜에 대해서만 통제가 가능하다.
	- TCPWrapper를 사용하면 **/etc/inetd.conf** 파일 안 해당 서비스의 실행경로에 **/usr/sbin/tcpd**를 명시한다.
	- 이후 외부로부터 서비스 요청이 올 경우 inetd.conf 파일을 참조하여 실행경로에 설정된 /usr/sbin/tcpd(TCP Wrapper 프로세스)를 실행한다.
	- tcpd는 hosts.allow, hosts.deny 파일을 참조하여 접근제어를 수행한 후 실행 인수로 설정된 서비스를 실행한다.
- **(나) hosts.allow(접근 허용)와 hosts.deny(접근 금지)**
	- 접근허용 및 차단에 관한 판단은 **/etc/hosts.allow**와 **/etc/hosts.deny**로 한다. 이를 통하여 외부로부터 접근을 선택적으로 제한할 수 있다.
		- 먼저 hosts.allow 파일을 참조하여 호스트 정보가 있다면 접근을 허용한다.
		- 다음 hosts.deny 파일을 참조하여 정보가 있다면 접근을 차단한다.
		- 두 파일 둘 다 없다면 default로 모든 접근을 허용한다.
	- 두 파일 모두 접근하고자 하는 해당 클라이언트가 설정사항에 포함되어 있지 않으면 접근을 허용하게 된다.
	- 설정 형식은 `[데몬 목록] : [클라이언트 목록] : [셸 명령어]`
		- 데몬 (서비스) 목록, 클라이언트 목록은 예약어 사용이 가능하며 각 구분자 `,`로 구분 가능하다.
		- 2줄 이상으로 기술할 경우 역슬래시를 이전 줄 마지막에 입력함.
		- 예약어는 다음과 같다.
			- ALL : 모든 서비스나 호스트
			- LOCAL : 같은 네트워크 안에 있는 모든 호스트
			- B EXCEPT A : 리스트 B에서 A를 제외한 모든 B
	- **예시**
		- `[hosts.allow] ALL : 192.168.9.0/255.255.255.0` : 192.168.9.0 네트워크 대역에 속한 모든 클라이언트 접속을 허가함.
		- `[hosts.deny] ALL : ALL`  : 모든 서비스에 대해서 모든 호스트의 접근을 차단한다.
		- `[hosts.allow] in.telnetd : .aaa.co.kr` : aaa.co.kr 도메인이 속한 모든 호스트는 Telnet 서비스가 가능하다.
		- `[hosts.allow] ALL EXCEPT in.telnetd : ALL` : 모든 호스트에 대해 텔넷을 제외한 모든 서비스가 가능하다.
	- 일반적인 설정이 hosts.allow 파일에 허용할 호스트를 명시한 후 hosts.deny 파일에 그 외의 모든 호스트를 차단하도록 설정한다. 먼저 처리되는 hosts.allow 파일에 허용하지 않는 호스트들은 전부 차단하게 됨.
	- 설정에 오류가 없는지 확인하기 위해 **/usr/sfw/sbin/tcpdchk**를 실행해본다. 오류가 출력되지 않으면 설정 파일에 문법적인 오류는 없는것.

#### 3) PAM(Pluggable Authenticiation Modules)
- **(가) 개요**
	- 리눅스 배포판에서 사용자 인증의 핵심.
	- 각 응용 프로그램에 대한 인증 형태, 사용자 권한, 접근 자원 등을 선택할 수 있는 라이브러리.
	- 시스템 관리자는 다양한 인증 서비스를 선택할 수 있고 기존 응용 프로그램을 수정할 필요 없이 새로운 인증 서비스 모듈을 추가하여 사용할 수 있는 프로그램이다.
- **(나) PAM을 사용한 인증 절차**
	- 각 프로그램(login, Telnet, FTP 등)은 인증이 필요한 부분에 PAM 라이브러리를 호출한다. 
	- PAM 라이브러리가 호출되면 해당 프로그램의 PAM 설정파일을 참조하여 등록된 여러 PAM 모듈들을 수행하고 그 결과를 응용 프로그램에 반환한다.
	- 응용 프로그램은 그 반환된 결과를 이용하여 인증 여부를 결정한다.
- **(다) PAM 설정 파일**
	- 설정 형식은 아래와 같다.
		- `[타입] [행동] [모듈 경로] [모듈 인자]`
		- 타입 : 모듈의 종류를 의미한다.
			- account, auth, password, session이 있다.
			- account는 계정 관련 전부.
			- auth는 패스워드 검증이나 다른류의 검증, 인증.
			- password는 비밀번호 인증 외 다른 비밀번호 관련.
			- session은 인정 처리 전후에 수행할 작업을 지정.
		- 행동 : 각 모듈 실행 후 성공 또는 실패에 따른 PAM 라이브러리 행동을 결정한다.
			- requisite, required, sufficient, optional이 있다.
			- requisite은 모듈 실행에 실패하면 인증 결과를 PAM 응용 프로그램에 통지하고 바로 인증을 거부하도록 한다.
			- required는 모듈 실행에 실패하더라도 즉시 인증을 거부하지 않고 동일 유형 모듈 실행을 완료한 시점에서 인증을 거부한다.
			- sufficient는 이전 모듈이 실패하더라도 이 모듈에서 인증에 성공하면 PAM은 인증을 승인하고 실패하면 평가를 계속한다.
			- optional은 모듈의 성공, 실패 응답을 상관하지 않는다. 거의 사용하지 않음.

## 5. 리눅스(Linux)
#### 1) 리눅스의 개요 및 특징
- 유닉스를 PC 버전으로 개발하려 노력한 산물
- 리누즈 토발즈가 개발함
- 현재 리눅스는 리처드 스톨만에 의해 FSF의 GNU 프로젝트로서 진행되고 있음.
	- FSF는 Free Software Foundation의 약자로 모든 소프트웨어는 자유롭게 공유되어야 한다는 취지에 따라 활동하고 있는 비영리단체.

#### 2) 파일시스템의 종류 및 특징
- 리눅스는 다양한 파일시스템을 지원한다.
	- minix : 과거 미닉스에서 사용되었던 파일 시스템으로 가장 오래되고 기본이 되는 파일 시스템. 파일시스템은 30문자로 제한되며 파일시스템마다 최대 64메가의 성능 제한이 있다.
	- ext : 리눅스 초기의 파일 시스템
	- ext2(Second Extended File System, 2차 확장 파일 시스템)
		- ext3가 나오기 전까지 대부분의 리눅스에서 채택됨
		- 캐시에 저장되어 있는 데이터를 디스크로 저장하는 도중에 만약 시스템이 다운되거나 여러 문제가 발생할 경우 파일 시스템이 손상되는 단점을 가짐.
		- 때문에 ext2는 fsck라는 파일 시스템 복구 기능을 제공한다. 하지만 복구하는데 시간이 오래 걸리고 복구시간 동안 시스템 사용이 불가능함.
	- ext(extended file system 3)
		- 커널 2.4버전부터 지원하고 있는 파일 시스템
		- ext2의 단점을 보완한 ext3 파일 시스템으로 뛰어난 무결성, 복구 기능을 가짐.
		- ext2 파일 시스템과 호환되며 저널링 기술을 채택하였다.
