## 1. 서버관리자의 업무
### (1) 보안 유지 관리
#### 1) 개요
- 시스템을 적절하게 구축하여 보호하고 배치했다면 지속적인 보안 유지 관리를 해야한다.
	- 로깅 정보를 감시하고 분석하기
	- 정기적 백업, 시스템 보안 테스트하기

#### 2) 로깅
- 로깅은 이미 발생한 나쁜 일을 알려주는 반작용적 제어. 
- 하지만 로깅은 시스템의 파괴, 고장 시에 무슨 일이 발생했는지를 빠르고 정확하게 찾을 수 있도록 도와주고 효율적으로 교정과 복구 노력에 집중할 수 있게 해준다.
- '로깅은 온전한 보안 자세의 초석이다.'

### (2) 서버관리자의 업무
#### 1) 시스템관리자 계정으로 직업하기
- 시스템 관리자로 작업하는 방법에는 두 가지가 있다. 
	- 루트로 로그인하기
	- 일반 사용자로 로그인 후 su 명령으로 계정을 바꾸기
- **루트로 로그인하기**
	- 계정이 루트로 뜨며, 프롬포트 모양이 `#`이다.
- **su를 사용해 루트 계정으로 변경하기**
	- 보통 직접 로그인하는걸 막아두고 일반 사용자로 로그인한 후 su 명령을 통해 관리자 계정으로 변경하는 것이 좋다.

#### 2) 사용자 계정 관리
- 리눅스는 멀티태스킹의 특징을 가지고 있으므로 다중 사용자를 수용하여 작업 수행이 가능. 
- 사용자마다 계정을 생성하면 /etc/passwd 파일에 계정, 패스워드 사용자 및 그룹 그리고 셸이 정의된다.
	- /etc/passwd는 일반 사용자도 접근하여 파일 내용을 볼 수 있다. 때문에 shadow 패스워드를 사용하여 /etc/shadow에 암호문으로 저장한다.
	- /etc/shadow 파일은 계정명, 암호화된 패스워드 등 9개의 필드로 되어있어 일반사용자에게는 접근권한 자체가 없으므로 안전하다.
- 계정 사용을 제한하려면 두 가지가 가능하다.
	- /etc/passwd 파일에서 /bin/bash 같은 셸을 삭제하여 원격 접근권한을 제거하기
	- /etc/shadow 파일에서 사용기간이나 만료일을 설정하여 계정 사용기간을 설정하기.

#### 3) 자원 관리
- **프로세스 관리**
	- ps, kill, wait, su 등을 사용한 프로세스 관리
	- wait : 프로세스가 끝나기를 기다리는 명령어.
	- nice : 프로세스 우선순위를 변경하는 nice 값을 설정하는 명령어
- **메모리 관리**
	- free : 시스템의 실제 메모리와 스왑 메모리에 대한 사용 현황을 확인할 수 있는 명령어
- **메일, 디스크 등의 자원 관리**
	- 리눅스 시스템의 경우 디스크 사용량을 사용자 및 그룹별로 설정하기 위해서 quota를 사용하며, 전체적인 메일 용량 크기의 제한은 메일 스풀 디렉터리에 별도의 파티션을 주고 그 파티션에 쿼터를 설정함.
	- du : 디스크의 파일 사용량을 재귀적으로 보여줌.

#### 4) 네트워크 관리

| 구분            | 설명                                       |
| ------------- | ---------------------------------------- |
| ifconfig      | 통신 디바이스(NIC) 상태를 보여준다.                   |
| netstat -an   | 현 시스템에서 사용되는 통신 서비스의 상태를 보여준다.           |
| top           | 시스템 자원(CPU, Memory 등)의 사용 현황을 보여준다.      |
| ps -elf(-aux) | 현 시스템에서 수행 중인 프로그램과 데몬 상태를 보여준다.         |
| who, w        | 로그인되어 있는 사용자를 보여준다.                      |
| snoop         | 네트워크를 흐르는 패킷을 캡처하여 분석한다.                 |
| nslookup      | 도메인에 대한 IP 정보 및 도메인 네임과 관련된 여러 검색이 가능하다. |
| hostname      | 현 시스템에 할당된 호스트 이름을 보여준다.                 |
| traceroute    | 특정 호스트까지 네트워크 라우팅 경로 및 경유하는 IP를 보여준다.    |
| inetd         | 네트워크 슈퍼데몬인 inetd를 실행한다.                  |

## 2. 로그 설정과 관리
### (1) 시스템 로그 분석
#### 1) 로그에 대한 이해
- 윈도우는 이벤트라는 중앙 집중화된 로그를 수집하여 저장하지만 유닉스는 로그를 여러 곳에 산발적으로 저장함.
- 윈도우는 로그를 중앙 집중화하기 때문에 관리가 편하지만, 공격자가 한 로그만 삭제하면 되니 위험하고 로그에 대한 보안 수준도 낮다.
- 하지만 유닉스는 초보자의 경우 로그를 찾는 것조차 쉽지 않으며 공격자가 로그를 모두 찾아 지우기 쉽지 않다.

#### 2) 윈도우의 로그 분석과 설정
- **(가) 해커에 대한 즉각적인 확인**
	- 시스템에 남은 로그 정보도 중요하지만 현재 로그인된 사용자를 확인하는 것도 중요함. 윈도우에서 이를 확인할 수 있는 명령은 `net session`이다.
	- 로그인한 시스템 IP, 계정, 클라이언트 운영체제, 세션의 수, 로그인한 후 경과한 시간을 출력한다. 이런 세션을 끊는 명령은 `net session /delete`이다.
	- 시스템에 현재 로그인한 세션을 알아보는 또 다른 툴로는 `psloggedon`이 있다. 이 툴은 현재 로컬로 로그인한 계정 정보도 함께 보여준다. 원격에 있는 관리자가 로컬로 로그인한 사용자 정보를 필요로 할 때 유용하다.
- **(나) 윈도우 시스템 이벤트 로그 종류**
	- 응용 프로그램, 보안, 시스템 로그 총 3가지로 나뉘고 이벤트에 기록한다.
	- **응용 프로그램 로그**
		- 응용 프로그램이 기록한 다양한 이벤트가 저장됨. 기록되는 이벤트는 개발자에 의해 결정됨
		- AppEvent.Evt
	- **보안 로그**
		- 유효하거나 유효하지 않은 로그인 시도 및 파일 생성, 열람, 삭제 등의 리소스 사용에 관련된 이벤트를 기록한다. 
		- 감사로그 설정을 통해 다양한 보안 이벤트 저장이 가능하다.
		- SecEvent.Evt
	- **시스템 로그**
		- 윈도우 시스템 구성요소가 기록하는 이벤트. 
		- 시스템 부팅 시 드라이버가 로드되지 않는 경우와 같이 구성요소의 오류를 이벤트에 기록한다.
		- SysEvent.Evt
	- **디렉터리 서비스 로그, 파일 복제 서비스 로그**
		- 디렉터리 서비스 로그는 윈도우 액티브 디렉터리 서비스에서 발생하는 이벤트이다. 
			- 예 : 서버와 글로벌 카탈로그 사이의 연결 문제
		- 파일 복제 서비스 로그는 윈도우 파일 복제 서비스에서 발생하는 이벤트이다.
			- 예 : 도메인 컨트롤러가 시스템 볼륨 변경 정보로 업데이트되고 있는 동안 발생하는 파일 복제 실패
		- 둘 다 도메인 컨트롤러 구성시에 작성됨.
	- **DNS 서버 로그**
		- 윈도우 DNS 서비스에서 발생하는 이벤트이다.
			- DNS 서버 구성시에 작성됨.
- **(다) 감사 정책**
	- 어떤 로그를 남길지 정의한 규칙이다.
	- 설정하게 되면 감사 정책에 의해 지정한 이벤트 범주에 대해서만 로그가 남음.
	- **감사 정책 종류**
		- **개체 액세스 감사**
			- 권장값 : 감사없음
			- 특정 파일이나 디렉터리, 레지스트리 키, 프린터 등과 같은 객체에 대하여 접근을 시도하거나 속성 변경 등을 탐지한다.
		- **계정 관리 감사**
			- 권장값 : 실패
			- 신규 사용자, 그룹의 추가, 기존 사용자 그룹의 변경, 사용자의 활성화나 비활성화, 계정 패스워드 변경 등을 감시한다.
		- **계정 로그인 이벤트 감사**
			- 권장값 : 성공, 실패
			- 계정 로그인 이벤트 감사는 도메인 계정으로 사용으로 생성됨. 그냥 로그인 이벤트 감사는 로컬 계정의 사용으로 생성됨.
		- **권한 사용 감사**
			- 권장값 : 실패
			- 권한 설정 변경이나 권한이 필요한 작업을 수행할 때 로깅한다.
		- **로그인 이벤트 감사**
			- 권장값 : 성공, 실패
			- 로컬 계정의 접근 시 생성되는 이벤트를 감사한다. 계정 로그인 이벤트 감사에 비해 다양한 종류의 이벤트를 확인할 수 있다.
		- **디렉터리 서비스 액세스 감사**
			- 권장값 : 실패
			- 시스템 액세스 제어 목록(SACL)이 지정되어 있는 액티브 디렉터리 개체에 접근하는 사용자에 대한 감사로그를 제공.
		- **정책 변경 감사**
			- 권장값 : 성공, 실패
			- 사용자 권한 할당 정책, 감사 정책 또는 신뢰 정책의 변경과 관련된 사항을 로깅함.
		- **프로세스 추적 감사**
			- 권장값 : 감사없음
			- 사용자 또는 응용 프로그램이 프로세스를 시작하거나 중지할 때 해당 이벤트가 발생한다.
		- **시스템 이벤트 감사**
			- 권장값 : 감사없음
			- 시스템의 시동과 종료, 보안 로그 삭제 등 시스템의 주요한 사항에 대한 이벤트를 남긴다.
- **(라) 로그 정책 설정**
	- 윈도우 로그 정책은 `제어판 - 관리도구 - 로컬 보안 정책 메뉴`를 선택하여 `로컬 정책 - 감사 정책`에서 확인할 수 있다. 
	- 윈도우의 경우 유닉스에 비해 로깅하는데 시스템 자원이 많이 소모되므로 모든 정보를 로깅하도록 설정하는 것은 바람직하지 않음,
- **(마) 이벤트 로그 파일**
	- 이벤트 로그 파일은 `%Windows%\system32\config`폴더 아래에 있는 .evt 확장자를 가진 파일이다. 
	- 파일 명은 이벤트 뷰어에서 보이는 로그 파일명과 비슷함.
		- 응용 프로그램 로그 : AppEvent.Evt
		- 보안 로드 : SecEvent.Evt
		- 시스템 로그 : SysEvent.Evt
	- 이멘트 로그 파일은 바이너리 형식이기 때문에 그냥 봐서는 그 의미를 알 수 없고 논리적 구조를 분석할 수 있는 별도의 프로그램으로 열어야 한다. (이벤트 뷰어)
	- `제어판-관리도구-이벤트 뷰어`를 통해 쌓이는 로깅 정보를 확인 가능.
	- 이벤트 뷰어에 표시된 종류, 날짜, 시간 등의 항목은 직관적이다.
	- 이벤트 표시 옆의 숫자는 고유 ID.
	- 이벤트 뷰어에 표시되는 내용은 다음과 같다.
		- 종류 : 정보(Information), 경고(Warning), 오류(Error), 성공 감사(Success Audit), 실패 감사(Failure Audit)가 있다. 성공 감사는 감사대상으로 설정된 대상에 대한 접근 시도가 성공했을 때, 실패 감사는 접근 시도가 실패했을 때 남기는 로그.
		- 날짜, 시간 : 로그를 남긴 날짜와 시간
		- 원본, 범주 : 로그와 관계 있는 영역
		- 이벤트 ID : 윈도우에서는 각 로그별로 고유한 번호를 부여함. 로그를 분석할 때 이 번호를 알고 있으면 빠르고 효과적인 분석이 가능하다.
		- 사용자, 컴퓨터 : 각각 관련 로그를 발생시킨 사용자와 시스템을 의미.

#### 3) 유닉스/리눅스의 로그 분석과 설정
- **(가) 로그 디렉터리**
	- 유닉스에서는 로그를 중앙 집중적으로 관리하는 syslogd 데몬이 있지만 각 데몬이 로그를 별도로 남기는 경우도 많음. 
	- 로그가 저장되는 경로는 시스템마다 다르고 두 개 이상의 경로에 로그를 나누어 저장하기도 함.
	- 주요 디렉터리 위치는 다음과 같다.
		- **/usr/adm**
			- 초기 유닉스, BSD 계열 : HP-UX, SunOS 4.x
		- **/var/adm**
			- 최근 유닉스, SVR 계열 : 오라클 솔라리스, HP-UX 10.x 이후, IBM AIX
		- **/var/log**
			- 일부 BSD 계열: BSD, FreeBSD, 오라클 솔라리스, 리눅스
		- **/var/run**
			- 일부 리눅스
- **(나) utmp(x) 로그**
	- utmp(x) 파일에 로그를 남기는 프로그램은 umtp 데몬.
	- 유닉스의 가장 기본적인 로깅을 제공하는 데몬.
	- 현재 시스템에 로그인한 사용자의 상태를 출력한다.
	- utmp 파일에서는 utmp.h에서 정의된 구조체로 로그인 데이터를 저장한다. 이때 텍스트가 아닌 바이너리 형태이기 때문에 vi 에디터로 확인이 불가능.
	- 특정 명령을 통해 확인이 가능한데, utmp 데몬에 저장된 로그를 출력하는 명령은 다음과 같다.
		- w, who, users, hwodo, finger 등
- **(다) wtmp(x) 파일**
	-
