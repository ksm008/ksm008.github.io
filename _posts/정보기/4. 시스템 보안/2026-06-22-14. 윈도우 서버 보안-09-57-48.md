## 1. 윈도우
#### 1) 윈도우 시스템의 구조
- **(가) 개요**
	- 윈도우 운영체제의 구조는 링 기반으로 5개의 링으로 구분됨.
	- 가장 하단에 하드웨어.
	- HAL(Hardware Abstraction Layer), 마이크로 커널, 각종 관리자 응용 프로그램 순으로 나뉨.
	- 윈도우에서는 하드웨어에서 하드웨어를 제어하기 위한 HAL, 마이크로 커널, 각종 관리자로의 연결을 커널 모드로 구분하고 마지막 링을 사용자 모드로 구분한다.
	- 드라이버 개발자와 하드웨어 개발자는 윈도우에서 제시한 기본 표준만 따르면 되는데 그 표준이 HAL.
		- HAL은 개발된 소프트웨어 간에 원할한 통신이 가능하도록 도와주는 번역자 역할을 함.

#### 3) 윈도우 파일 시스템
- **(가) FAT16**
	- FAT는 기본적으로 2GB까지가 한 파티션. FAT 테이블의 기본 크기는 16비트인데, 이는 하드디스크에 파일을 담는 최소 단위인 클러스터가 32KB이기 때문.
	- 따라서 가능한 용량이 2GB가 되는 것. (`2^16 * 32 * 1024 = 2 *1024 * 1024 * 1024 = 2GB`)
	- 처음에는 충분해 보였지만 이후 기술의 발전으로 작은 용량이 되어버려 FAT32가 나옴.
- **(나) FAT32**
	- 윈도우 95 OSR(OEM Service Release)2에서 처음 도입된 FAT32는 FAT와는 비교할 수 없을만큼 용량이 커져 총 32비트(2^32)개의 클러스터를 가질 수 있다.
	- 현재도 사용됨. 때문에 호환성이 좋아 리눅스나 다른 운영체제에 정보를 옮길 때 유용하다. 하지만 접근제어 설정이 불가능 해 보안과는 거리가 먼 파일 시스템.
- **(다) NTFS**
	- 접근제어가 적용된 윈도우 파일시스템에는 NT 계열에 적용되기 시작한 NTFS(New Technology File System)가 있다.
	- NTFS는 파일시스템의 결함 관리(HotFixing)에 있어 FAT나 FAT32보다 훨씬 좋은 성능을 보인다. 보안 관점에서도 비교적 훌륭함.
	- 개별 폴더와 파일에 사용 권한을 설정할 수 있고 각 파일과 폴더의 해당 계정에만 접근하여 읽을 수 있게 암호화할 수 있기 때문. 
		- 폴더를 암호화하면 폴더에 있는 파일도 모두 암호화되며, 암호화된 폴더나 파일은 복사나 이동을 해도 암호화 속성을 잃지 않는다.
	- NTFS는 특징 계정의 사람이 몇 날 몇 시 몇 분에 파일을 열거나 열려다 실패했는지 알 수 있는 감사(Auditing) 기능을 제공한다.
	- NTFS의 디스크 구조는 다음과 같다.
		- **MBR(Master Boot Record)**
			- 파티션 생성 시 물리적 디스크의 첫 번째 섹터에 위치하는 512-Byte 크기의 영역
			- 부트 코드, 파티션 테이블, 시그너처로 구성되어있음. 
			- 운영체제가 부팅될 때 BIOS에 의해 POST(Power On Self Test) 과정을 마친 후 MBR의 부트 코드를 호출하고 부트 코드는 부팅 가능 파티션 (Active 파티션)을 찾아 해당 파티션 VBR의 부트 코드를 호출하여 운영체제가 기동하게 됨.
		- **VBR(Volume Boot Record)**
			- 윈도우 부팅을 위한 기계어 코드와 볼륨 및 클러스터의 크기, MFT의 시작주소 등 설정 정보를 담고 있으며 파티션의 가장 마지막 섹터에 VBR 백업본을 가지고 있다.
		- **MFT(Master File Table)**
			- 볼륨/파일 시스템에 존재하는 모든 파일과 디렉터리에 대한 정보를 담고 있는 테이블.
			- NTFS에서 가장 중요한 영역. 
			- 하나의 파일이나 디렉터리에 대한 정보를 담고 있는 MFT 엔트리 집합으로 구성됨.
				- 파일의 위치, 시간 정보, 크기, 파일 이름 등 다양한 속성 정보들로 구성되어 있음.
		- **시스템 파일**
			- MFT를 비롯해 파일 복사나 삭제 과정 중 시스템이 오류를 일으킬 때가 있는데 이를 복구하는데 사용함.
			- 디스크 사용에 대한 로그 파일, 디스크 볼륨 이름 등 디스크 자체에 대한 정보를 담는 파일이 저장됨.
		- **파일 영역**
			- 각 파일에 대한 실제 데이터가 저장됨.
- **(라) 디스크 파괴형 악성코드의 특징**
	- 디스크 파괴형 악성코드는 마스터 부트 레코드(MBR) 또는 볼륨 부트 레코드(VBR) 영역을 훼손하여 컴퓨터가 정상적으로 부팅되지 않도록 하는 악성코드를 말한다. 예를 들어 MBR 영역의 모든 값을 특정 문자나 쓰레기 값으로 덮어씌워 운영체제 부팅을 할 수 없도록 하거나 VBR 영역을 훼손하여 파일시스템 복구가 불가능하도록 만든다.
	- MBR 영역이 손상된 경우에는 부트 코드를 정상적인 디스크에서 복사하고 각각의 파티션 정보를 수집하여 파티션 테이블을 완성함으로써 복구가 가능함.
	- VBR 영역이 손상된 경우에는 파티션의 가장 마지막 섹터에 있는 VBR 백업본을 덮어써서 복구가 가능함.
- **(마) FAT16과 FAT32 파티션을 NTFS로 변환**
	- 하드디스크 성능을 최적화하려면 NTFS 파일 시스템이 거의 유일한 선택
	- 보통 NTFS가 16이나 32보다 성능이 좋음. 특히 큰 파티션에 파일이 많이 있을 경우 더더욱 그렇다.
	- NTFS 성능을 최대로 끌어내려면 파티션을 NTFS로 포맷하고 파일을 저장해야 하지만 여의치 않은 경우에는 CONVERT 유틸리티를 사용하여 변환도 가능하다.

#### 4) 윈도우 부팅 순서
- **(가) 윈도우 XP, 윈도우 서버 2000/2003의 부팅 순서**
	- **1단계 : POST(Power On Self Test) 실행**
		- POST는 운영체제와 관련된 부분은 아니지만 어떤 운영체제를 설치해도 POST 과정을 거친다.
		- POST는 하드웨어 스스로 시스템에 문제가 없는지 기본 사항을 검사하는 과정.
	- **2단계 : 기본 부팅 관련 설정사항 로드**
		- BIOS는 CMOS(Complementary Metal-Oxide Semiconductor)에 설정되어 있는 시스템 설정사항 및 부팅과 관련된 여러 가지 정보를 읽어 시스템에 적용한다.
	- **3단계 : MBR(Master Boot Record, 마스터 부트 레코드) 로드**
		- MBR은 저장 매체의 첫 번째 섹터(LBR 0)에 위치하는 512 바이트의 영역으로 부팅 매체에 대한 기본 파일 시스템 정보가 들어있다. 
	- **4단계 : NTLDR (NT Loader) 실행**
		- 하드디스크의 부팅 파티션에 있는 프로그램. 
		- 윈도우 서버 2000이 부팅될 수 있도록 간단한 파일시스템을 실행하고 boot.ini 파일의 내용을 읽어 가능한 부팅 옵션을 보여준다.
	- **5단계 : NTDETECT.com 실행**
		- NTDETECT는 NTLDR에서 제어권을 위임받아 시스템에 설치된다. 
		- 이후 다음과 같은 하드웨어 유형을 검사한다.
			- PC의 CPU
			- 버스
			- 비디오 보드
			- 키보드와 마우스
			- 직렬/병렬 포트
			- 플로피 드라이브
	- **6단계 : ntoskrnl.exe(NT OS Kernel) 실행**
		- ntoskrnl은 HAL, DLL을 로드한다.
- **(나) 윈도우 비스타, 2008, 7의 부팅 순서**
	- **1~3단계 : POST 실행/기본 부팅 설정사항 로드/MBR 로드**
		- 윈도우 XP나 서버 2000/2003의 부팅 순서와 동일.
	- **4단계 : 윈도우 부트 서브 시스템 실행 (Window Boot Manager) 실행**
		- 윈도우 7은 MBR에서 NTLDR이 실행되지 않고 윈도우 부트 서브시스템이 실행된다. 윈도우 부트 서브 시스템에서는 bootmgr.exe가 실행되고 부트 설정 데이터(BCD, Boot Configuration Data)를 읽어 실행 가능한 운영체제의 목록을 보여준다.
	- **5단계 : 윈도우 로더(Winload.exe) 실행**
		- NTDETECT와 같이 각종 드라이브를 로드하고 ntoskrnl.exe를 실행한다.

## 2. 윈도우 보안
### (1) 윈도우 설치
#### 1) 윈도우 설치 시 중요사항
- **(가) 파티션 나누기**
	- 하나의 물리적인 하드디스크에 논리적으로 분할 영역을 만드는 것을 말한다.
	- 하드디스크에 칸막이 공사를 하여 하나의 하드를 별개의 하드디스크로 쓰기 위함.
	- 파티션을 나누면 서로 다른 드라이브로 인식하기 때문에 C에서 오류가 발생해도 D는 안전하다.
	- 또한 윈도우를 2개 이상 설치하여 사용하는 멀티부팅도 사용이 가능하고 중요 데이터 저장 목적으로 파티션을 나누는 경우도 있다.

### (2) 윈도우 계정, 권한, 인증
#### 1) 윈도우의 계정과 권한 체계
- **(가) 기본 사용자와 그룹**
	- 윈도우에 생성되어 있는 사용자는 `제어판 - 컴퓨터 관리 - 로컬 사용자 및 그룹`메뉴에서 확인할 수 있다.
	- 윈도우를 설치할 때 기본적으로 생성되는 계정을 정리하면 다음과 같다.
		- Administrator : 관리자 권한 계정. 사용자가 사용 가능한 계정 중 가장 강력함.
		- SYSTEM : 시스템에서 최고 권한을 가진 계정. 관리자보다 권한이 높음. 원격 접속이 불가능하고 이 계정을 사용하여 로그인 불가.
		- Guest : 매우 제한적인 권한을 가진 계정. 기본 설정은 사용 불능이다.
	- **+ 운영체제별 관리자 계정**
		- 유닉스 : root
		- 윈도우 : Administrator
		- NetWare : Supervisor
	- 기본적으로 생성되는 그룹들은 다음과 같다.
		- Administrator : 모든 권한이 다 있음.
		- Account Operator : 사용자나 그룹 계정을 관리하는 그룹
		- Backup Operator : 시스템 백업을 위해서 모든 시스템의 파일과 디렉터리에 접근할 수 있음.
		- Guests : 도메인 사용 권한이 제한된 그룹, 시스템의 설정 변경 권한이 없음.
		- Print Operators : 도메인 프린터에 접근할 수 있다.
		- Power Users : 디렉터리나 네트워크 공유, 공용 프로그램 그룹 생성, 컴퓨터의 시계 설정 권한 등이 있다.
		- Replicator : 도메인에 있는 파일을 복제할 수 있는 권한을 가진 그룹. 로컬 로그인과 시스템 재시작 및 종료 권한이 있다.
		- Sever Operators : 도메인의 서버를 관리할 수 있는 권한을 가진 그룹. 로컬 로그인과 시스템 재시작 및 종료 권한이 있다.
		- Users : 도메인과 로컬 컴퓨터를 일반적으로 사용하는 그룹. 개개인에 할당된 사용자 환경을 직접 만들 수 있지만, 설정 항목에 한계가 있다. 시스템 서비스 시작 및 종료 권한이 없으며 디렉터리 공유 설정이 불가능하다.
- **(나) SID(Security Identifier)**
	- 윈도우의 각 사용자나 그룹에 부여되는 고유한 식별번호.
	- 유닉스/리눅스 시스템의 UID, RUID, EUID와 유사하게 계정을 하나의 코드 값으로 만든것.
	- 윈도우에서는 whoami /user 명령을 통해 로그인 한 사용자의 SID를 알아볼 수 있다.
	- 예시 : `S-1-5-21-4198389319-964410498-2111126834-1001`
	- SID를 분석하면 다음과 같다.
		- S : SID를 의미
		- 1 : revision number (SID 명세 버전)
		- 5 : Identifier authority value(48비트, 윈도우 보안 권한)
		- 21-4198389319-964410498-2111126834
			- Sub-authority value로 도메인 또는 로컬 컴퓨터 구분자라고도 한다.
			- 시스템의 고유한 숫자. 시스템을 설치할 때 시스템의 특성을 수집하여 생성된다. 동일한 컴퓨터에 다시 윈도우를 설치해도 동일한 값을 가지지 않음.
		- 1001
			- relative ID라고 한다.
			- 기본적으로 생성되는 빌트인 계정이 아니라면 1000번보다 큰 숫자가 생성된다.
			- 관리자는 500, Guest는 501, 일반 사용자는 1000번 이상이다.
	- SID는 로컬에서만 사용하는 것이 아니다. 시스템 A에서 B로 관리자 계정이 로그인을 시도할 때 A의 관리자 계정과 시스템 B의 계정은 다르기 때문에 SID로 구별한다.
	- Well-Known SID 목록
		- S-1-0-0 :  SID를 모를 때 사용하는 SID
		- S-1-1-0 : Everyone
		- S-1-5-7 : Anonymous
		- S-1-5-18 : System Profiles(시스템의 서비스용 계정)
		- S-1-5-19 : Local Service
		- S-1-5-20 : Network Service
		- S-1-5-domain-500 : Administrator
		- S-1-5-domain-501 : Guest

#### 2) 윈도우의 권한 상승
- 관리자나 시스템으로 실행되고 있는 프로세스의 권한을 빼앗는 걸 권한 상승이라고 한다.
- 프로세스의 권한을 빼앗는 방법은 기본적으로 상위 권한으로 수행되고 있는 프로그램의 프로세스에 다른 작업을 끼워넣는 것이다.
	- 관리자 A와 일반 사용자 B가 있다.
	- A가 P라는 프로그램을 실행하면 p1, p2라는 프로그램 상의 실행 절차를 거쳐 프로그램이 완료된다. 
	- 여기에 일반 사용자 B가 p1을 q1으로 바꿔 놓고 자신이 P를 실행하여 생성한 상위 권한 프로세스를 가로채는 것.

#### 3) 윈도우 인증과 패스워드
- **(가) 윈도우 인증과 구성요소**
	- 윈도우의 인증과정에서 가장 중요한 구성 요소는 LSA(Local Security Authority), SAM(Security Account Manager), SRM(Security Reference Monitor)이다. SAM은 윈도우에서 패스워드를 암호화하여 보관하는 파일의 이름과 동일함.
	- **LSA**
		- 모든 계정의 로그인에 대한 검증을 진행하고 시스템 자원(파일 등)에 대한 접근 권한을 검사한다. 이름과 SID 매칭, SRM이 생성한 감사 로그를 기록하는 역할도 함. 
		- LSA는 NT 보안의 중심 요소. 보안 서브시스템이라 불린다.
	- **SAM**
		- 사용자 그룹 계정 정보에 대한 데이터베이스를 관리한다. 
		- 사용자 로그인 입력 정보와 SAM 데이터베이스 정보를 비교해 인증 여부를 결정한다. 
		- 윈도우의 SAM 파일은 다음 경로에 위치함.
		- systemroot는 윈도우가 설치된 폴더를 의미함. 주로 `C:/Windows`이다.
		- `%systemroot%/system32/config/sam`
		- SAM이 사용자의 계정과 패스워드의 일치 여부를 확인하여 SRM에 알리면 SRM은 사용자에게 SID를 부여한다.
	- **SRM**
		- SID에 기반하여 파일이나 디렉터리에 대한 접근을 허용할지를 결정하고 이에 대한 감사 메시지를 생성한다.
- **(나) SAM 파일 접근 통제 설정**
	- SAM 파일은 사용자와 그룹 계정의 패스워드를 관리하고 LSA를 통한 인증을 제공하는 중요한 파일이므로 적절한 접근 통제가 가능.
	- 공격자로부터 SAM 파일에 대한 패스워드 공격 시도에 따른 정보 노출의 위험이 있다.
	- 보안설정
		- 관리자 및 시스템 그룹 외에는 SAM 파일에 대한 접근을 제한함.
		- 불필요한 그룹이나 계정에 대해서는 권한을 제거함.

### (3) 공유자료 관리
#### 1) 네트워크 드라이브의 이해
- 대상 컴퓨터의 드라이브를 내 컴퓨터에서 네트워크 드라이브로 설정하여 내 컴퓨터의 드라이브처럼 사용할 수 있는 기능.
- 내 컴퓨터 - 우클릭 - 네트워크 드라이브 설정 클릭
- cmd에서 `net use 드라이브명: \\ip\설정대상드라이브$`를 실행하고 계정 및 패스워드를 입력하면 설정 가능.

#### 2) 파일과 폴더의 보안권한 설정
- 윈도우는 폴더에 6가지 기본 권한설정을 제공하며 파일에는 5가지 권한을 제공한다.
	- **공통 권한**
		- 모든 권한 : 폴더에 대한 접근 권한과 소유권 변경 가능. 서브 폴더와 파일을 삭제 가능.
		- 수정 : 폴더 삭제 가능. 읽기 및 실행과 쓰기가 동일한 권한.
		- 읽기 및 실행 : 파일의 내용을 읽고 실행할 수 있지만 수정하거나 삭제할 수 없다.
		- 읽기 : 디렉터리의 내용을 읽기만 가능함.
		- 쓰기 : 디렉터리의 서브 디렉터리와 파일을 생성할 수 있고 소유권이나 접근 권한의 설정 내용 확인이 가능
	- **폴더 권한**
		- 폴더 내용 보기 : 디렉터리의 파일이나 서브 디렉터리 이름을 볼 수 있음.
- **표준 권한 할당하기**
	- 윈도우는 GUI를 통해 접근제어리스트를 관리함.
	- 파일/폴더에 우클릭 - 속성 - 보안 - 편집 - 사용권한 대화상자

#### 3) 디렉터리 및 파일에 대한 접근 권한 설정
- 윈도우의 NTFS에서 설정한 디렉터리 및 파일에 대한 접근 권한 설정은 다음 규칙을 갖는다.
	- **규칙1. NTFS 접근 권한은 누적된다**
		- 개별 사용자가 여러 그룹에 속한 경우 특정 파일이나 디렉터리에 대한 접근 권한은 누적된다.
		- 예를 들어 그룹 A에서는 읽기 권한, 그룹 B에서는 쓰기 권한을 할당 받았다면 읽기와 쓰기 둘 다 권한이 있는 셈.
	- **규칙2. 파일에 대한 접근 권한이 디렉터리에 대한 접근 권한에 우선한다.**
		- 파일이 포함된 디렉터리의 권한보다 파일에 대한 권한 설정이 우선한다.
	- **규칙3. 허용보다 거부가 우선**
		- 윈도우에서는 유닉스와 달리 허용 권한이 없는 것은 거부의 의미가 아니다. 윈도우는 허용과 거부 중 하나를 선택해야 한다.
		- 만약 중첩 권한 중 명백한 거부 설정이 있다면 허용보다 거부가 우선하여 적용됨.

#### 4) 공유 자료 숨기기
- **(가) 공유 폴더 숨기기**
	- 공유 리소스를 설정할 때 이름 끝에 $ 기호를 붙이면 된다.
	- 드라이브 L:을 공유하려고 할 때 공유 이름을 L$로 사용하면 됨. 이렇게 하면 네트워크 창에서 원격 컴퓨터를 열 때 리소스의 목록에 나타나지 않음. 
	- 숨겨진 공유 폴더 보기 : cmd에 `net share` 입력
	- `C$, D$, ADMIN$`공유가 숨겨졌더라도 매우 잘 알려져 있어 침입자가 네트워크에 액세스할 수 있는 위험이 있다. 따라서 비활성화 해야됨
	- 단순히 공유를 중지시켰다고 해서 기본 공유 기능이 중지된 것은 아님. 왜냐면 운영체제 재부팅시 다시 자동으로 기본 공유 폴더를 만들기 때문에...
	- 완전히 제거하려면 레지스트리 편집기를 열어 `\HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\services\LanmanServer\parameters`에서 DWORD 값을 0으로 새로 만든다.
- **(나) 윈도우 공유 폴더**
	- **C$ 혹은 D$ 등**
		- C드라이브에 대한 관리 목적 공유 폴더. 드라이브가 많을수록 더 표시됨.
		- 기본 공유 폴더를 통해서 인가받지 않은 사용자가 하드디스크 내의 폴더나 파일에 접근 가능.
		- 보안적으로 상당히 위험한 상황이 기본 공유 폴더를 통해 발생할 수 있으며 기본 공유 폴더를 이용하여 바이러스가 침투한 사례도 있음.
	- **ADMIN$**
		- 윈도우 설치 폴더에 접근하는 관리목적 공유 폴더. 이 공유로 접근하면 그 컴퓨터의 윈도우 설치폴더가 열리게 된다. 
		- 윈도우 버전마자 윈도우 설치 폴더가 다르므로 모든 윈도우를 대상으로 어떠한 파일을 복사하거나 변경할 필요가 있을 때 주로 사용함.
	- **IPC$(Inter Process Communication)**
		- 프로세스간 통신이라는 뜻으로 풀이됨. 네트워크 등에서 프로세스 간 통신을 위해서 사용하는 통로.
		- IPC는 동시에 수행될 개별 프로그램을 생성하고 다룰 수 있게 해주는 인터페이스다.
		- 단일 사용자의 요구가 운영체제에서 여러 프로세스를 수행하는 결과를 가져올 수 있기 때문에 이러한 사용자를 위해 프로세스 간 통신이 필요하며 IPC 인터페이스가 이를 가능하게 함.
		- IPC$를 제거하면 네트워크 서비스 일부에 문제가 발생할 수 있으므로 레지스트리 값을 수정해서 익명 사용자의 네트워크 접근이 불가능하도록 설정하는 것이 좋다.

### (4) 암호 기능 사용
#### 1) 폴더 및 파일 암호화(EFS)
- EFS(Encrypting File System)은 윈도우 운영체제(Windows XP 및 Windows Server 2000 이상)에서 개별 파일 또는 특정 폴더 안에 들어 있는 모든 파일을 암호화할 수 있는 기능을 제공함.
- 파일이나 폴더의 속성 부분을 변경하여 사용할 수 있으며 인증서 추가/삭제를 통해 EFS가 적용된 파일을 읽을 수 있는 사용자 추가/변경이 가능함.

#### 2) 볼륨 암호화(BitLocker)
